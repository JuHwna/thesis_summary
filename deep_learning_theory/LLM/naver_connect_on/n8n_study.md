- 예시) 고객 문의를 자동으로 회신하고 또 직원에게 알람으로 갈 수 있는 자동화 플랫폼의 workflow 예시

    - 긴급한 내용이 아니면 바로 처리 로그로 가서 고객이 물어봤던 질문들을 상세적으로 기록할 수 있는 부분으로 기록해주고 여기에 접수되었습니다까지 완성하는 흐름

- 노드 연결 및 노드이 돌아가는지 설명
  - (1) 메시지를 받는 노드
    - 고객이 메시지를 이렇게 입력하게 된다라고 했을 때 그 고객 메세지를 받아서 우리 LLM 텍스트 모델이 받아오게 됨
  - (2) LLM 텍스트 모델에서 받아오는 노드
    - 물어봤던 질문들을 그대로 프롬프트를 받아 오게 됨
    - 질문을 다양하게 분류할 수 있는 프롬프트를 미리 입력해두어야 함
      - 카테고리를 어떻게 나눌 것이냐
      - 질문을 그대로 넘겨주는게 아니라 질문 받아온 내용을 정리할 수 있도록 카테고리로 나눔
      - 얼마나 급한지 그 정도도 한 번 나눠봐서 급하다고 하면 슬랙에 알람이 갈 수 있도록 설정
      - 분류 기준에 대한 설정도 설명을 해놓음
        - 설명해줬을 때 이게 어떤 카테고리에 소속되는지 구분해놓음
      - 최종 출력 형태도 넣음
    - 왼쪽은 INPUT, 오른쪽은 OUTPUT
    - Gemini : 무료로 쓸 수 있는 api기능들, 무료로 쓸 수 있는 사용량도 제공해줌
  - (3) 텍스트를 파싱하는 부분
    - 앞의 output 값들이 이번엔 input으로 가져오게 됨
    - output은 조금 더 정리한 형태로 나오게 됨
      - 구분해서 나올 수 있록 처리를 해주는 부분
  - (4) 분류하는 노드
    - input: 방금 가져온 내용들을 체계에 맞게 정리를 해둠
    - 급한 부분에 대한 구분값을 구한 후에 어떤 구분값일 때 슬랙에 알람을 가게 할 것인지를 정하는 단계
  - (5) 구글 시트에 내용 적힘
  - (6) 고객에게는 문의 접수 알림 보이게 됨


## 1. 문제 정의와 업무 자동화 기획
### 1. [이론] 시스템적 사고 및 아웃풋 중심의 기획법
#### 문제 해결 접근법
##### 문제 해결 접근법의 개요
- 왜 기획이 먼저인가?
  - 자동화 도구를 배우기 전에 기획 능력을 먼저 갖춰야 하는 이유
  - 자동화는 만드는 기술이 아니라 설계하는 사고에서 시작함
    - 도구는 사고를 실행하는 수단일 뿐
  - 우리가 무엇을 만들지 왜 만드는지 어떤 결과가 나와야 하는지를 미리 정하지 않았기 때문
  - 자동화는 기능을 많이 알고 있는 사람이 잘하지 X
    - 문제를 정확하게 정의하고 결과를 먼저 설계한 사람이 잘함
  - 도구를 배우기 전에 어떻게 생각하는지부터 정리가 필요함 
- 기획 없이 만들면
  - 노드를 연결했는데 기대한 대로 작동이 안 됩니다.
  - 자동화를 완성했는데 실제로 쓸 수가 없습니다
  - 어디서부터 시작해야 할지 감이 안 잡힙니다
  - 열심히 만들었는데 처음부터 다시 해야 합니다
- 기획을 먼저하면
  - 어떤 노드가 필요한지 미리 파악할 수 있음
  - 무엇을 만들어야 하는지 명확하고 첫번째 버전부터 맞는 방향의 결과물이 나옴
  - 수정과 디버깅 횟수가 줄어듬
  - 협업할 때에도 설명이 쉬워짐

- 이런 경험, 한 번쯤 있지 않으신가요?
  - 열심히 만들었는데 결과가 기대와 다를 때 문제는 대부분 기획에 있음
    - 일단 노드를 연결해보자 -> 어 생각한대로 안 되네-> 처음부터 다시 만들어야 하나
    - 자동화를 만들긴 했는데 쓰다 보니 빠진 게 너무 많아
    - 상사에게 보여줬더니 내가 원한 게 이게 아닌데라고 했어
    - 워크플로우가 가끔씩 멈추는데 언제 멈추는지조차 모르겠어.
  - 이런 문제가 생기는 이유 3가지
    - (1) 최종 결과물을 먼저 정의하지 않아서 어디로 가는지 모르고 달리기 시작한 것과 같음 => 방향을 잃은 상태
    - (2) 전체 시스템의 흐름을 파악하지 않아서 나무만 보고 숲 전체를 그리지 못한 상태 => 부분은 맞은데 전체는 안 맞는 상태
    - (3) 피드백 루프를 고려하지 않아서 한 번 실행되고 멈추는 자동화, 오류가 나도 해결하기 어려운 상태
    - 도구의 문제가 아니라 생각하는 방식의 문
- 만들기 전에 두 가지 사고법으로 먼저 설계해야 함
  - **아웃풋 중심 사고**와 **시스템 사고**

- 아웃풋 중심 사고 vs 시스템 사고
  - 두 사고법은 서로 다른 질문에 답하며 함께 쓸 때 가장 강력함

|구분|(1) 아웃풋 중심 사고|(2) 시스템 사고|
|----|-------------------|----------------|
|핵심 질문|"최종적으로 어떤 결과물이 나와야 하는가?"|"전체 시스템이 어떻게 순환하며 작동하는가?"|
|사고 방향|결과물->거꾸로->과정 설계(역방향)|요소->프로세스->피드백->개선(순환)|
|주요 목표|전달 시점의 아웃풋을 완성도 있게 만들기|지속적으로 스스로 개선하는 시스템 만들기|
|잘 어울리는 상황|보고서 생성, 알림 발송, 문서 처리 자동화|데이터 모니터링, 반복 수집분석, 지속적 운영|
|n8n 활용|어떤 데이터가 최종적으로 어디에 전달될지 먼저 설계|워크플로우가 반복 순환하며 자기 교정하도록 설계|

##### part1. 아웃풋 중심 사고
- 원하는 결과물을 먼저 머릿속에 그린 다음, 거꾸로 과정을 설계함
- 아주 단순한 질문에서 시작함
  - 최종적으로 어떤 결과가 나와야 하는가?
  - 많은 분들이 일을 시작할 때 무엇부터 해야 되지?
  - 아웃풋 중심 사고는 그 반대로 시작함
- 1.[개념이해] 아웃풋 중심 사고란 무엇인가? 왜 이 방식이 효과적인가?
- 2.[4단계 프레임워크] 문제 재정의 -> 아웃풋 이미지 -> 성공 조건 -> 프로세스
- 3.[적용 예시] 워크플로우 기획에 어떻게 적용할까?

##### 문제 해결 접근법: (1) 아웃풋 중심 사고
- 아웃풋 중심 사고란 : '전달 시점의 아웃풋'을 먼저 상상하며 일하는 방식
- 일을 시작하기 전에 최종 결과물이 어떤 모습이어야 하는가를 먼저 구체적으로 그린 다음, 그것을 만들기 위해 거꾸로 과정을 설계하는 사고 방식
- 스스로에게 어떤 결과가 나와야 하는지, 누가 어디서 어떤 형태로 이 결과를 보게 될 것인
  - 이 질문이 정리된 다음에 그 결과를 만들기 위한 필요 과정들을 설계함 
<img width="1417" height="383" alt="image" src="https://github.com/user-attachments/assets/b1b88825-9215-408c-b2d2-6a6125139d1f" />

- 아웃풋 중심 사고의 4단계 프레임워크
- (1) 문제 재정의 : 이 자동화가 해결하려는 진짜 문제는?
  - 요청받은 상황을 내 언어로 다시 이해하는 단계
  - 표면적 요구 뒤에 숨은 진짜 필요를 파악함(진짜 불편함을 찾는 것)
- (2) 아웃풋 이미지 : 최종 결과물은 구체적으로 어떤 모습?
  - 누가, 무엇을, 어디서, 어떤 형태로 받아보게 되는지 최대한 구체적으로 그림
- (3) 성공 조건 : 잘 작동했다고 볼 수 있는 기준은?
  - 이 워크플로우가 성공적으로 작동했다고 판단할 수 있는 세부 조건들을 목록으로 만듦
    - ex) 시간이 얼마나 줄어야 할지, 빠진 정보는 없는지, 문제가 생기면 알 수 잇는지, 이런 기준들을 목록으로 만드는 단계
    - 이 단계가 나중에 테스트할 때 하나의 기준이 되기도 함
- (4) 프로세스 : 어떤 노드들이 어떤 순서로 동작?
  - 앞 3단계를 모두 정한 뒤에야 노드 구성과 데이터 흐름을 설계함
  - 자동화 도구로 옮기기 시작함 
- 주의 : (1)->(2)->(3)->(4) 순서로 생각해야 함
  - 4번부서 시작하게 되면 방향 없이 노드를 쌓게 됨

##### part 2. 시스템 사고
- 전체 시스템이 어떻게 순환하며 작동하는지 파악함 - 지속적인 변화를 만드는 사고 방식
  - 계속 안정적으로 돌아가게 하는 사고방식
- 1.[개념이해] 순환 프로세스와 피드백이란? 아웃풋 중심 사고와 무엇이 다른가?
- 2.[3가지 구성 요소] 요소(Element) - 프로세스(Process) - 피드백(Feedback)
  - 실제로 무엇이 있고 어떻게 흘러가고 문제가 생기면 어떻게 대응하는지 정리하는 방식
- 3.[적용 예시] 지속적으로 스스로 작동하는 워크플로우를 어떻게 설계할까?
  - 관점을 자동화에 적용해 보는 것
  - 자동화가 멈추지 않게 하고 문제가 생기면 알아차리고 필요한 조치를 할 수 있도록 지속적으로 돌아가는 구조 설계

##### 문제 해결 접근법: (2) 시스템 사고
- 시스템 사고란 무엇인가? : 지속적인 변화를 일으키는 순환 프로세스를 파악하는 사고 방식
- 어떤 시스템이 왜 이렇게 작동하고 어떻게 변화하며 어떻게 지속되는가를 요소들 사이의 순환 관계와 피드백으로 파악하는 사고 방식
  - 전체 흐름과 반복 구조를 이해하는 것
  - 계속 잘 되게 잘 돌아가게 만드는 것
  - 순환 : 일이 한 번 끝나면 그냥 끝나는 게 아니라 계속 반복되면서 돌아가는 구조
  - 피드백 : 그 과정에서 문제가 생기거나 결과가 나왔을 때 다음에 어떻게 조정할지를 알려주는 신호
|구분|아웃풋 중심|시스템 사고|
|----|----------|------------|
|언제|전달 시점|지속적으로|
|목표|아웃풋 완성|지속적 변화|
|수단|단방향 프로세스|순환 프로세스|

- 아웃풋 중심 사고 : 결과를 완성하는 사고
- 시스템 사고 : 그 결과가 계속 나오도록 유지하게 하는 사고

- 시스템의 3가지 구성 요소
- 요소(element) : 시스템을 이루는 부품들
  - 시스템에 영향을 주는 하위 항목들
    - 시스템을 구성하는 각각의 독립적인 부분이라고 생각함
  - 예시 : RSS 피드 소스, AI 요약 모델, Slack 채널, 에러 로그 저장소, 이메일 알림 등
- 프로세스(Process) : 요소들이 서로 영향을 주고 받는 방식
  - 하위 요소들 사이에서 목표 달성을 위해 영향을 주고받는 과정
    - 요소들을 연결하는 다리
  - 예시 : 트리거 -> RSS 수집 -> 키워드 필터 -> AI 요약 -> Slack 발송
- 피드백(Feedback) : 결과를 보고 다시 조정하는 루프
  - 목표에 기반하여 하위 항목들을 리뷰하고 교정하는 과정
    - 시스템을 살아있게 만드는 핵심
  - 예시 : 오류 감지 -> 재시도 or 담당자 알림, 주간 성공률 리포트 생성

- 세 요소의 관계 : 요소들이 프로세스를 통해 연결되고 피드백이 시스템 전체를 목표 방향으로 유지, 개선함

### 2. [이론] 업무 자동화 프로세스 이해 및 기획
#### 업무 자동화 프로세스 이해 및 기획하기
##### 우리에게 더 중요한 것은 무엇일까요?
- 어떤 자동화 도구가 생겨도 변하지 않는 핵심 능력이 존재
- 도구는 문제 해결의 수단
  - 어떤 도구를 쓰든, 문제를 정의하고 구조를 설계하는 능력이 자동화 역량의 본질 
- 특정 툴을 잘 쓰는 것?
  - n8n, Zapier, Make 등 도구는 계속 바뀜
  - 지금 잘 쓰는 도구가 내년에 사라질 수도 있음
- AI 개념을 이해하는 것?
  - LLM, RAG, 에이전트 등 개념 이해도 중요하지만 그 자체가 목적이 될 수는 없음
- 문제를 잘 해결하는 것
  - 어떤 문제인지 정의하고 어떤 구조로 풀지 설계하는 능력이 가장 오래 남음
  - 자동화의 본질 : 어떤 문제를 해결할 것인지 명확히 하고 그 문제를 어떤 흐름으로 풀 것인지 설계하는 힘
    - 도구는 수단일 뿐

##### 업무 자동화 기획 - 4단계 프로세스
- 아이디어 발굴에서 실제 서비스까지, 체계적으로 접근하는 방법
- 자동화를 한 번 만드는 방법이 아니라 업무에서 계속 돌아가게 만드는 방법을 배우는 과정 
- (1) 아아디어 선정
  - 어떤 문제를 자동화할지 발굴하고 선택하기(실제로 효과가 큰 문제 선택)
- (2) 아웃풋 기획
  - AI 솔루션의 구조와 결과물 형태 설계하기
  - 무엇을 만들 것인지 먼저 정하기
    - 결과물이 어떤 형태이고 누구에게 전달되고 어떤 흐름으로 작동할 것인지 구조를 잡는 단계
- (3) 빠른 제작
  - 기획을 토대로 워크플로우를 빠르게 구현하기
  - 완벽하게 만들기 보다는 동작하는 첫 버전을 빠르게 구현하는 것
- (4) 실전 적용
  - 배포 후 반복 개선하며 지속 운영하기

##### Part 1. 아이디어 선정

- 1.[문제 발굴] 현장에서 겪는 실제 페인포인트 찾기
- 2.[기준으로 필터링] 임팩트 + 기술 타당성으로 평가하기
- 3.[하나 선택] 가장 좋은 하나에 집중하기

- SET 노드 - 데이터 필드 조작의 스위스 칼
  - 간단한 값을 변경하고 필드를 추가하는 데 적합
  - 복잡한 것들만 활용하면 점점 더 자동화가 복잡해질 수 있어서 가능하면 SET 노드를 조금 더 활용하는데 집중해야 함
  - set 노드를 더 많이 쓰면 복잡성을 떨어뜨릴 수 있음
