# Chapter 05. 오케스트레이션
- 오케스트레이션 : 단순히 어떤 도구를 언제 호출할지 결정하는 것을 넘어 각 모델 호출에 적합한 컨텍스트를 구성해 효과적이고 근거 있는 행동을 유도하는 과정을 포함
  - 복잡한 워크플로 : 각 단계를 정확하게 수행하기 위해 정교한 계획, 메모리 검색, 동적 컨텍스트 조합이 필요함
  - 시스템이 보유한 리소스를 활용해 사용자 요청을 효과적으로 처리하도록 하는 핵심 로직
  - 사용자 요청을 처리하는 파운데이션 모델, 추가 정보를 검색하는 외부 및 로컬 도구, 다양한 데이터베이스 호출을 중심에서 제어하는 로직

## 5.1 에이전트 유형
- 에이전트 유형의 선택 : 시스템의 성능, 비용, 기능을 좌우함

### 5.1.1 반사형 에이전트
- 반사형 에이전트 : 입력과 행동을 직접 연결하며 내부 추론 과정을 거치지 않음
  - '조건이 참이면 도구 실행' 규칙을 따르며 사전에 정의된 트리거가 감지되면 즉시 적절한 도구를 호출함
  - 중간 추론 단계를 생략하므로 지연시간이 매우 짧고 성능이 예측 가능해 키워드 기반 라우팅, 단일 단계 데이터 조회, 간단한 자동화 같은 용도에 적합
  - 단점 : 표현력이 제한적이어서 다단계 추론이나 입력 외부의 컨텍스트를 요구하는 작업은 처리할 수 없음

### 5.1.2 리액트 에이전트
- 리액트 에이전트 : 추론과 행동을 반복 루프 내에서 교차 수행함
  - 모델이 사고를 생성하고 도구를 선택해 호출한 뒤 결과를 관찰하고 다시 계획을 갱신하는 식
  - 복잡한 작업을 여러 관리 가능한 단계로 분할하고 중간 결과를 반영하며 계획을 조정할 수 있게 함
  - 강점 : 탐색적 시나리오(ex 동적 데이터 분석, 다중 소스 통합, 문제 해결)에 강하며 중간에 전략을 조정할 수 있는 유연성 장점
    - 도구 호출, 관찰 결과, 단계별 의사결정 등 실행 과정의 흔적이 로그로 남기 쉬워 디버깅과 감사에 유리함
  - 단점 : 루프 구조로 인해 API 비용과 응답 시간이 늘어날 수 있음
 
### 5.1.3 계획 후 실행 에이전트
- 계획 후 실행 에이전트 : 작업을 두 개로 뚜렷한 단계로 나뉨
  - 계획 단계 : 모델이 다단계 계획을 생성
  - 실행 단계 : 이어서 계획된 각 단계를 도구 호출을 통해 수행
  - 역할을 분리하면 계획 단계에서 장기적인 추론에 집중하고 실행 단계는 필요한 툴만 호출해 불필요한 LLM 호출을 줄임
    - 계획이 명시적으로 드러나기 때문에 디버깅과 모니터링도 쉬워짐
    - 생성된 계획을 점검하고 어느 단계에서 실패했는지 추적한 뒤 필요하면 다시 계획을 세울 수 있음
  - 장점
    - 명확한 분해 : 복잡한 작업을 관리 가능한 하위 작업들로 나눌 수 있음
    - 디버깅 용이성 : 계획이 명시적이어서 오류가 어디서, 왜 발생했는지 드러남
    - 비용 효율성 : 실행은 더 작은 모델이나 더 적은 LLM 호출로 처리하고 큰 모델은 계획 수립에만 집중시킬 수 있음
