# Chapter 03. 에이전트 시스템을 위한 UX 디자인
- 사용자 경험(UX) : 성공을 좌우하는 핵심 요소
- 에이전트의 효과성, 신뢰성, 도입을 결정하는 건 => 사용자가 에이전트와 상호작용하는 방식
  - 파운데이션 모델과 에이전트 아키텍처가 뛰어난 기술적 능력을 구현함

- 에이전트 모달리티 유형 비교

|모달리티|보편성|예시 활용 사례|적합한 상황|
|-------|------|-------------|-------------|
|텍스트|매우 흔함|고객 지원 챗봇, 생산성 어시스턴트|명확하고 비동기적이며 검색 가능한 소통이 필요할 때|
|그래픽 사용자 인터페이스(GUI)|흔함|워크플로 오케스트레이션 대시보드, 커서 같은 ai코딩 어시스턴트|시각적 구조, 컨텍스트 관리, 다단계 워크플로가 중요한 경우|
|음성|덜 흔함|시리, 스마트 홈 어시스턴트(알렉사, 구글 홈), 콜센터 자동화|손을 쓰지 않는 상호작용이나 자연스러운 대화가 필요한 경우|
|비디오|드물|가상 튜터, 의료 상담 아바타, 대화형 학습 에이전트|시각적 시연, 풍부한 표현, 몰입형 학습이 필요한 경우|

- 핵심 UX 고려 사항
  - 시간이 길어질 경우 컨텍스트를 어떻게 관리하느냐임
  - 에이전트의 능력, 한계, 불확실성을 명확히 알리는 일은 현실적인 기대치를 설정하고 오해를 방지하는 데 필수
- 신뢰와 투명성은 에이전트 시스템의 긍정적 사용자 경험을 떠받치는 토대


## 3.1 모달리티
- 텍스트 기반 인터페이스 : 명료성, 추적성이 뛰어남
- 그래픽 인터페이스 : 풍부한 시각 정보와 직관적 제어 제공
- 음성 인터페이스 : 손을 쓰지 않아도 편한 편의성을 제공함
- 비디오 인터페이스 : 역동적이고 실시간에 가까운 소통을 가능하게 함
- 에이전트 시스템에서 뛰어난 UX를 제공하기 위한 핵심 강점, 과제, 모범 사례를 검토

### 3.1.1 텍스트 기반 인터페이스
- 에이전트 시스템과 상호작용하는 가장 흔하고 범용적인 방식
  - 채택되는 이유 : 단순성, 친숙함, 기존 워크플로에 쉽게 통합되는 점
  - 장점 : 동기식(실시간) 대화와 비동기식 상호작용을 모두 지원
    - 사용자가 나중에 돌아와도 컨텍스트를 잃지 않고 이어갈 수 있음
    - 텍스트 상호작용은 교환 내역이 명확하게 남음
    - 문제가 생겼을 때 투명성, 책임성, 트러블 슈팅이 쉬워짐
    - 정밀성, 추적성, 비동기 소통이 중요한 상황에서 빛을 발함
      - 생산성 도구에서 강력함
    - 구조화된 데이터베이스에서 특정 질문에 답하거나 데이터를 조회하는 지식 검색 시스템에서도 효과적
    - 접근성과 배포 용이성 덕분에 에이전트 UX 설계의 초석이 됨
   
- 사례
  - 텍스트 기반 모달리티
    - 워프 : 자연어 명령 변황, 지능형 자동완성, 컨텍스트 기반 설명을 통합해 개발자 터미널을 재해석함
      - 커맨드라인을 협업 중심의 ai 증강 작업 공간으로 바꿈
      - 자연어 프롬프트로 복잡한 문법이나 플래그를 외우지 않고도 명령을 생성, 실행, 디버깅할 수 있음
    - 클로드 코드, 제미나이 CLI : 자연어 상호작용을 코드 생성, 실행, 파일 조작으로 확장함
      - 커맨드라인 전문지식이 있는 사람만 쓰던 도구가 초보자와 전문가 모두에게 자연어로 시스템과 상호작용한는 접근 가능한 강력한 통로로 바뀌고 있음
      - ai 터미널 : 대화형 파트너로서 사용자 의도를 해석하고 모범사례를 제안, 실시간으로 오류를 디버깅함
  - 슬랙, 팀즈, 왓츠앱 같은 메시징 앱의 채팅 인터페이스

- 핵심 한계
  - 발견 가능성
    - 사용자가 에이전트가 지원하는 능력이나 효과적인 명령 표현 방식을 모르는 경우가 많음
    - 제공 기능을 추측하거나 기억해야 함
      - **행동 유도성**이 부족하면 혼란이 생기고 에이전트 기능이 충분히 활용되지 않으며 요청이 지원 범위를 벗어날 때 사용자의 불편함이 커집
  - 자연어의 모호성
    - 예상치 못한 방식으로 요청 표현 가능(견고한 의도 인식 필요)
    - 응답 길이 제약의 영향
    - 감정적 뉘앙스 한계

- 효과적인 텍스트 기반 에이전트 설계
  - 발견 가능성을 높이는 전략이 필요
    - 온보딩 메시지, 주기적 기능 안내, 대화 중 동적 제안을 통해 지원 기능을 선제적으로 알려야 함
  - 명확성, 컨텍스트 유지, 오류 관리에 대한 세심한 주의 필요
    - 명확성 : 간결하게 소통(과도한 전문 용어와 장황한 설명은 피해야 함)
    - 컨텍스트 유지 : 대화가 길어지면 컨텍스트를 유지 중요
    - 오류 관리 : 문제 상황에도 우아하게 대응 필요
      - 명확한 오류 메시지와 폴백 메커니즘을 제공함

### 3.1.2 그래픽 인터페이스
- 여러 그래픽 요소를 결합해 사용자와 에이전트 시스템이 시각적인 상호작용을 하도록 함
  - 효과적인 곳 : 시각적 명료성, 구조화된 워크플로, 다단계 프로세스가 필요한 작업
  - 장점
    - 정보를 시각적으로 제시해 인지 부하를 줄인다는 점

- 사례
  - 대시보드 기반 ai 도구, 그래픽 채팅 인터페이스, 클릭 가능한 요소를 갖춘 에이전트 기반 생산성 플랫폼
  - 워크플로를 관리하는 에이전트
    - 대시보드로 대기 작업, 완료 단계, 오류 알림을 보여줘 사용자가 시스템 상태를 한눈에 파악하도록 할 수 있음
    - Langsmith, n8n, Arize, AutoGen
    - 최신 AI 지원 IDE 인터페이스 : 커서, 윈드서퍼, 클라인에 존재
      - 자연어 이해를 코딩 워크플로에 직접 통합해 개발자가 질문하고 코드를 생성 및 리팩터링하고 설명이나 성능 최적화를 받는 일을 하나의 간결한 그래픽 인터페이스 안에서 수행하게 함
  - 데이터 시각화, 구조화된 상호작용, 명확한 상태 업데이트가 필수인 사례에 강함
    - 작업 관리 대시보드, AI 에이전트가 구동하는 데이터 분석 도구, 필터와 시각적 미리보기를 갖춘 전자상거래 추천 시스템, 사용자 질문에 맞춰 구조화된 출력을 동적으로 생성하는 ui 시스템, 하이브리드 워크플로
  - 러버블, 커서, 윈드서퍼, 깃허브 코파일럿 같은 도구 => 고품질 GUI 대거 선보임

- 새롭게 떠오르는 분야: 생성형 UI 등장
  - 정적인 대시보드나 사전 설계도니 레이아웃에만 의존 x
  - 사용자 질의에 따라 인터페이스 요소, 데이터 시각화, 구조화된 출력을 동적으로 만듦
  - 자연어의 유연성과 그래픽 레이아웃의 명료성, 발견 가능성을 결합해 에이전트가 필요할 때마다 풍부하고 컨텍스트에 특화된 인터페이스를 만들어 줌
    - ex) 퍼플렉시티 AI : 텍스트 답변뿐 아니라 질문에 맞춘 지식 카드, 참고 목록, 데이터 테이블을 생성함
      - AI 코딩 코파일럿 : 사용자 의도를 기반으로 전체 폼, 설정 파일, UI 구성 요소를 생성함
  - 효과적이고 사용하기 쉬운 생성형 ui를 만들기 위해선 : 신중한 디자인 패턴, 레이아웃 제약, 우선순위 부여 로직이 중요함
  
- 단점
  - 화면의 물리적 공간이 한정적

- 효과적인 그래픽 에이전트 인터페이스 설계
  - 중요한 정보를 노이즈 속에 파묻지 않도록 우선순위를 정해야 함
  - 인터페이스 반응성을 관리
    - 비동기적 작동 시 : 실시간 업데이트와 상태 전환의 부드러움을 기대함
  - 화면 크기에 맞춰 적응해 일관성 유지 필요
  - 자동화와 사용자 통제의 균형
    - 에이전트가 캘린더 변경을 제안할 때 여러 옵션을 버튼으로 제시해 사용자가 최종 결정을 빠르고 명확하게 내리도록 할 수 있음

### 3.1.3 음성 인터페이스
- 사용자가 말을 주된 소통 방식으로 에이전트 시스템과 자연스럽고 손을 쓰지 않고 상호작용하도록 제공함
- 장점
  - 손을 쓰지 않아도 되는 편의성, 자연스러운 상호작용, 즉각적인 컨텍스트 반응성이 중요한 상황
  - 별도의 조작이 필요 없는 작업, 빠른 질의, 액션 중심 워크플로
  - 편의성, 접근성, 유연성 제공
- 단점
  - 조밀하거나 복잡한 정보를 다룰 때
    - 텍스트 인터페이스가 더 빠른 이해와 참조에 유리함
  - 조밀한 정보 소비나 빠른 훑어보기, 나란히 비교가 필요한 복잡한 의사 결정일 때
  - 


- 사례
  - 아마존 알렉사, 애플 시리 같은 가상 비서
  - 고객 지원 음성 봇
  - 시각 장애가 있거나 이동성이 제한된 사용자에게 접근 가능한 대안 제공


- 옛날 음성 인터페이스의 큰 장애물 : 지연 시간
  - 실시간으로 음성을 처리하려면 음성 인식, 의도 인식, 적절한 응답 생성 필요
    - 대화 흐름을 지연이 발생 => 인터페이스가 어색하거나 기계적으로 느껴짐

- 효과적인 음성 에이전트 인터페이스 설계
  - 중단을 우아하게 처리하는 능력..?
    - 유연성 반영 필요
      - 사용자가 명령을 중간에 멈춰도 혼란 없이 받아들이고 입력을 자연스럽게 수정하게 하며 대화를 새로 시작하지 않고 중단했던 지점에서 이어가야 함
  - 도구 사용
    -외부 컨텍스트 참조, 기록 갱신, 실시간 액션 수행 가능

- 인상적인 발전에도 불구하고 음성 인터페이스는 여전히 개척 단계

- 음성 인터페이스를 배포할 때
  - 사람이 음성과 문자를 처리하는 속도의 차이를 이해해야 함
    - 사람의 발화 속도 : 분당 150 ~ 180단어 정도
    - 읽기 속도 : 평균 분당 250 ~ 300단어
    - 훑어 보기 속도 : 500 이상


### 3.1.4 비디오 기반 인터페이스
- 시각, 청각, 때로는 텍스트 요소를 하나의 일관된 경험으로 결합하는 새로운 에이전트 상호작용 양식
- 줌, 마이크로소프트팀즈, 가상 이벤트 같은 플랫폼을 통해 비디오가 디지털 생활 전반에 퍼지면서 에이전트는 이러한 환경에 통합되는 새로운 방식을 찾고 있음
  - 장점
    - 시각적 단서, 음성, 텍스트 오버레이, 애니메이션 등 여러 감각 채널을 결합해 더 풍부하고 표현력 있는 상호작용을 제공한다는 점
    - 인간과 유사한 표정과 제스처를 모사해 소통에 감정적 뉘앙스를 더할 수 있음
    - 인간스러움을 더해 명료성, 감정적 연결, 전반적 효율을 높임
  - 단점
    - 불쾌한 골짜기
    - 기술적, 디자인적 과제
      - 고품질 비디오 상호작용에 상당한 연산 성능과 대역폭 필요
        - 랙이나 화질 저하를 유발해 사용자 경험을 해칠 수 있음
      - 표정, 제스처, 립싱크가 조금만 어긋나도 불쾌한 골짜기 위험이 커져 몰입보다 불편함을 줄 수 있음
    - 시각 데이터를 고유하는 데 대한 불안 => 프라이버시 우려 증폭
 
- 사례
  - 대면 대화를 모사하는 비디오 아바타부터 실시간 비디오 협업 도구에 내장된 에이전트까지 범위가 넓음
  - 원격의료, 교육, 원격 협업, 인터랙티브 엔터테인먼트 같은 산업
 

### 3.1.5 모달리티 결합을 통한 매끄러운 경험
- 여러 모달리티를 하나의 응집된 사용자 여정으로 결합함
- 사용자는 모발리티의 경계를 의식하지 않음
  - 가능한 수월하고 자연스럽게 목표를 달성하길 원함
- 훌륭한 에이전트 시스템 설계의 핵심 : 모달리티를 넘나들면서 상태와 컨텍스트를 유지하는 능력

- 모발리티 유동성 설계 시 필요 사항
  - 정보, 작업 진행 상황, 사용자 선호가 전환 과정에서 결코 유실되지 않아야 함
    - 상태 관리와 컨텍스트 지속성 필요
  - 모달리티에 맞게 커뮤니케이션 스타일을 조정해야 함
    - 음성 : 간결한 요약을 제공
    - 검토용 텍스트 출력 : 상세한 내용 제공
   

- 기술의 최전선이 빠르게 확장되더라도 핵심 UX와 제품 원칙은 변하지 않음

- 성공적인 에이전트 경험 구축
  - 최신 모달리티 통합이나 생성형 UI를 과시하는 데 있지 않음
  - 사용자를 깊이 이해하고 눈높이를 맞춰 신뢰할 수 있고 직관적이며 즐거운 경험을 제공해 실제 문제 해결하는 것
    - 사람들의 삶을 더 쉽게 만들어 주며 더 중요한 일에 집중하도록 돕는 도구를 만드는 것
   

### 3.1.6 자율성 슬라이더
- UX 설계 시 매우 중요한 차원 : 에이전트에 부여하는 자율성의 수준
- 안드레 카파시 설명
  - 효과적인 에이전트 시스템
    - 완전 수동 제어에서 부분 자동화, 완전 자율 운영까지 에이전트의 자율성을 사용자가 부드럽게 조정할 수 있어야 함
- 자율성 슬라이더
  - 개념: 사용자가 특정 시점에 유지할 통제와 위임할 범위를 선택할 수 있게 함
  - 유연성을 통해 시스템의 작동 방식을 사용자 선호, 작업 복잡도, 컨텍스트와 맞춰 사용자 신뢰를 구축함
  - 신뢰를 구축하는 메커니즘

- 사용자, 작업, 컨텍스트에 따라 요구되는 에이전트 자율성의 정도는 다름
  - 정밀함 필요 시 : 완전 수동 제어 선호
  - 반복적이거나 복잡한 작업 : 에이전트에 완전히 위임하기를 원할 수 있음
  - 중요한 점 : 선호가 고정되지 않다는 것 => 사용자 신뢰, 작업 친숙도, 위험도, 업무량에 따라 변화함
    - 수동 : 개발자가 에이전트 도움 없이 모든 코드를 직접 작성함
      - IDE: 문법 하이라이트와 린팅만 제공하는 편접기 역할을 함, AI 기반 제안은 없음
    - 요청(보조) : 에이전트가 코드 자동완성, 리팩터링, 문서 스니펫을 선제적으로 제안하지만 적용 전에는 개발자가 각 제안을 검토해 수락
      - 속도를 높이면서 개발자가 완전한 통제권을 유지함
    - 완전 자율 : 에이전트가 표준 리팩터링 적용, 린터 오류 수정, 프로젝트 규약에 따른 보일러플레이트 파일 생성 등 특정 작업을 개별 승인 없이 자율적으로 수행함
      - 개발자는 변경 사항을 알림으로 받지만 각 작업을 승인할 필요는 없음
    - 이 세 가지 모드 : 단일 인터페이스에서 개발자가 통제와 효율의 균형을 잡도록 도움

- 소프트웨어 개발을 넘어 다음과 같은 고객 지원 플랫폼에도 이 원칙 적용 됨
  - 수동 : 인간 상담원이 모든 고객 문의를 직접 처리함
    - AI는 비활성화되어 있거나 프런트라인 상호작용이 아닌 백엔드 분석에만 사용됨
  - 요청(보조) : 에이전트가 고객 메시지에 대한 제안 답변을 작성하고 권장 응답, 정책 참고 트러블슈팅 단계를 제시함
    - 인간 상담원이 답변을 검토하고 필요하면 수정한 뒤 발송을 승인함
    - 응답 속도를 높이면서도 인간의 판단을 유지함
  - 완전 자율 : 에이전트가 비밀번호 재설정, 주문 추적, FAQ 같은 일상 문의를 인간 개입 없이 자율적으로 처리하고 복잡하거나 민감한 이슈만 인간 상담원에게 전달함
    - 사용자는 에이전트의 조치에 대한 알림을 받지만 표준 상호작용마다 승인은 필요하지 않음

- 자율성 스펙트럼 : 에이전트 경험에 의식적으로 설계되어야 함
  - 그렇지 않으면 : 과도한 수동 입력 요구로 에이전트가 무력하게 느껴지거나 민감함 컨텍스트에서 사용자 동의 없이 행동해 과도하게 지배적으로 느껴질 위험이 있음

- 자율성 슬라이더를 효과적으로 통합하려면 다음 설계 원칙 고려
  - 자율성의 단계 노출을 명확히 하라
    - 수동에서 보조, 자율까지 가능한 에이전트 독립성 수준을 사용자가 이해해야 함
    - 수동, 보조, 자동처럼 직관적인 용어로 모드를 표시하고 각 모드의 의미를 설명함
  - 매끄러운 전환을 가능하게 하라
    - 사용자 신뢰, 컨텍스트, 업무량이 변활 때 자율성 수준을 손쉽게 바꿀 수 있어야 함
    - EX) 인터페이스의 토글이나 슬라이더로 검토 모드에서 자동 승인 모드로 빠르게 전환할 수 있어야 함
  - 각 수준에서 예측 가능하고 투명한 기능을 제공하라
    - 각 수준마다 작동 과정을 잘 정의해야 함
    - 보조 모드에서 출력 초안을 초안을 생성하되 실행 전 명시적 사용자 승인을 요구할 수 있음
    - 완전 자율 모드에서는 여전히 상태 업데이트와 개입 옵션을 제공해야 함
  - 각 수준의 위험과 이점을 소통하라
    - 자율성을 높일 때 얻는 이점과 감수하는 위험을 사용자가 인지해야 함
    - 중요 작업의 경우 완전 자율을 활성화하기 전에 명시적 확인을 요구하는 것이 바람직할 수 있음
  - 사용자 신뢰와 역량에 따라 자율성을 조절하라
    - 사용자가 에이전트를 사용하면서 신뢰를 쌓일수록 자율성 수준을 높이도록 제안할 수 있음
    
## 3.2 동기식 대 비동기식 에이전트 경험
- 동기식과 비동기식 디자인의 선택 : 작업의 성격, 사용자 기대, 운영 컨텍스트에 크게 좌우됨
- 동기식 에이전트 : 사용자와 실시간으로 상호작용하며 즉각적으로 대화를 진행함
  - 강점 상황 : 즉각적 피드백이나 실시간 의사결정이 필요한 작업
  - EX) 채팅 인터페이스, 음성 대화, 실시간 협업 도구에서 흔하며 흐름과 몰입을 유지하려면 빠른 응답이 필수
- 비동기식 에이전트 : 사용자와 독립적으로 작동하고 시간 간격을 두고 이따금씩 소통함
  - 강점 상황 : 시간이 오래 걸리거나 백그라운드 처리가 필요하거나 사용자의 지속적 주의를 요구하지 않는 워크플로에 더 적합함
  - EX) 이메일과 유사한 상호작용, 작업 알림, 처리 완료 후 전달되는 에이전트 생성 보고서

### 3.2.1 동기식 경험을 위한 디자인 원칙
- 동기식 에이전트 경험에서 중요한 요소 : 즉시성, 명확성, 반응성
  - 사용자는 에이전트가 신속히 응답하고 눈에 띄는 지연 없이 대화 흐름과 컨텍스트를 유지하기를 바람
- 동기식 상호작용 : 낮은 지연시간과 컨텍스트 인식 필수
  - 에이전트가 멈추거나 같은 질문을 반복하는 일을 피할 수 있음
- 우선해야하는 점
  - 응답에서 명료함과 간결함
    - 장황한 설명이나 과도하게 복잡한 출력 => 실시간 상호작용의 리듬을 깨뜨림
    - 턴 테이킹 관리 : 언제 응답하고 언제 기다리고 언제 요청을 전달할지
      - 자연스럽고 생산적인 대화 흐름을 유지하는 데 핵심임
  - 오류 처리 중요함
    - 출력이 잘못되거나 중단된다고 해도 상호작용을 망치지 않고 회복해야 함
    - 불확실성이 생기면 위험한 추정을 하기보다 명확화 질문을 하거나 부드럽게 방향을 전환해야 함
      - 불필요한 마찰 없이 사용자의 몰입을 유지하고 컨텍스트를 유지하는 매끄럽고 직관적인 경험을 만듦
     
### 3.2.2 비동기식 경험을 위한 디자인 원칙
- 비동기식 에이전트 경험에서 중요한 요소 : 유연성, 지속성, 장기적 명료성
  - 장시간 실행 작업을 처리하거나 상세 보고서를 준비하거나 백그라운드 이벤트를 모니터링하는 상황처럼 즉각 응답이 필요하지 않는 컨텍스트에서 자주 발생
- 효과적인 비동기식 에이전트
  - 작업 상태와 결과를 명확히 전달해야 함
    - 투명성을 유지하는 핵심 도구 : 알림, 요약, 구조화된 보고서
  - 사용자 기대를 효과적으로 관리해야 함
    - 명확한 일정, 진행 지표, 후속 알림 => 불확실성이나 에이전트 작업 가시성 부족으로 인한 좌절 예방

- 컨텍스트 관리 : 비동기식과 동기식 에이전트 모두 중요한 디자인 원칙
  - 에이전트는 과거 컨텍스트를 끊김 없이 보존하고 참조해야 함
  - 사용자가 진행 중인 작업으로 돌아올 때 정보를 반복하거나 이전 단계를 다시 밟지 않도록 해야 함

### 3.2.3 능동적 개입과 침해적 행동 사이의 균형
- 능동성 : 긴급 이슈 알림, 최적화 제안, 적시 리마인더 등에서 큰 도움이 됨
  - 타이밍이 안좋거나 귀찮은 알림은 사용자를 지치게 함, 워크플로를 방해, 심하면 완전히 이탈하게 만듦
- 능동성의 균형
  - 컨텍스트 인식과 사용자 통제에 달려 잇음
  - 에이전트는 사용자의 현재 집중 대상, 긴급도, 커뮤니케이션 선호를 이해해야 함
  - 에이전트가 능동적으로 연락할 때 관련성을 최우선해야 함
    - 알림과 제안은 문제를 해결하거나 인사이트를 제공해 도움이 되어야 함
    - 사용자는 알림 빈도, 채널, 전달 임계갓을 제어해야 함
- 균형은 기술적 역량만의 문제가 아님
  - 사용자의 워크플로와 심리 상태에 대한 공감이 필요함
 

## 3.3 컨텍스트 유지와 연속성
- 컨텍스트 유지와 연속성을 보장하는 것 : 효과적인 에이전트 시스템을 설계하는 데 중요
  - 컨텍스트를 유지하는 능력 : 사용성, 효율, 신뢰에 직접적인 영향을 줌
    - 기술적 능력이지만 UX 관점의 문제
      - 사용자가 에이전트를 통합적이고 주의 깊은 협력자로 느끼는지 아니면 반복을 강요하는 단절된 구로 느끼는지를 좌우하기 때문
      - 메모리 : 연속성, 개인화, 지능감을 제공함
        - 에이전트가 이전 상호작용, 사용자 선호, 진행 중인 작업을 기억하면 대화와 워크플로를 끊김 없이 이어 가며 인지 부하와 좌절을 줄임
    - 구현 방식은 UX를 직접 규정함
      - 순수 클라이언트 측 컨텍스트 : EX) 브라우저 메모리 저장
        - 세션 내에서 빠르게 느껴짐
        - 로그인 간 연속성을 잃음
      - 순서 서버 측 컨텍스트 : EX) 사용자 ID에 연결된 데이터베이스 저장
        - 장기 메모리와 크로스 디바이스 경험을 가능하게 함
        - 지연시간이나 프라이버시 고려 사항을 초래할 수 있음
      - 하이브리드 접근
        - 반응성을 위해 단기 컨텍스트는 클라이언트에 연속성을 위해 장기 컨텍스트는 서버에 보존
        - 대체로 최적의 UX 균형을 달성함
    - 효과적인 컨텍스 유지를 위해선
      - 단기와 장기 메모리를 모두 적절히 관리해야 함
      - 단기 메모리 : 진행 중인 세션에서 방금 전 질문의 세부나 지침을 유지하게 함
      - 장기 메모리 : 여러 세션에 걸쳐 신호, 과거 상호작용, 더 넓은 사용자 패턴을 보존해 시간이 지남에 따라 적응하게 함
    - 컨테스트 관리 : 신경 쓸 요소가 많음
      - 데이터 지속성, 프라이버시 문제, 메모리 한계를 신중히 다뤄야 함

### 3.3.1 상호작용 간 상태 유지
- 상태 관리 : 에이전트 시스템의 컨텍스트 연속성을 떠받치는 중요한 역할
  - 멀티턴 대화, 작업 인계, 중간 상태가 있는 워크플로에서 특히 중요함
  - 컨텍스트를 잃으면 좌절, 비효율, 작업 포기로 이어짐

- 효과적인 상태 관리 : 시스템이 사용자나 세션을 어떻게 식별, 추적하느냐에 달려 있음

- 에이전트 시스템이 수천, 수백만 사용자로 확장되면 세션 상태를 메모리에만 두어서는 안됨
  - 상태를 데이터베이스나 분산 캐시에 유지하면 서버 재시작 이후에도 연속성을 보장하고 로드 밸런싱을 지원하며 멀티디바이스 경험을 가능하게 함
  - 사용자 기반 메모리(지속적, 개인화)
  - 세션 기반 메모리(휘발적, 세션 범위)
  - 구현 방식과 무관하게 견고한 식별, 저장 전략 : 대규모에서 매끄럽고 컨텍스트 인식적인 경험을 제공하는 데 필수
 
- 효과적인 상태 유지
  - 명확한 세션 경계, 데이터 검증, 폴백 메커니즘을 요구함
  - 에이전트가 컨텍스트를 잊었다면 잘못된 추정 x, 명확화 질문으로 회복 필요
  - 보안과 책임 있는 관리가 필수

### 3.3.2 개인화와 적응성
- 개인화 : 단순한 컨텍스트 기억을 넘어 과거 상호작용과 선호를 사용해 에이전트의 행동, 응답, 추천을 사용자별로 조정함
  - 적응형 에이전트 : 상태를 유지하는데 그치지 않고 이전 교환에서 학습해 점점 더 정교하고 관련성 높은 결과를 제공함
  - 개인화 형태
    - 선호 보존 : 알림 선호, 자주 선택하는 옵션 같은 사용자 설정을 기억함
    - 행동 적응 : 관찰된 사용자 패턴에 기반해 응답 스타일이나 상호작용 흐름을 조정함
    - 능동적 지원 : 과거 행동을 바탕으로 사용자 필요를 예측하고 제안을 제공함
- 가장 이상적인 개인화
  - 에이전트가 조정했다는 사실을 내세우지 않으면서 경험을 자연스럽게 개선해 사용자는 기계적 도구가 아니라 사려 깊은 협력자로부터 이해 받고 지원받는다고 느껴짐

## 3.4 에이전트 능력 커뮤니케이션
- 효과적인 에이전트 경험을 설계할 때 중요한 과제 하나 : 사용자가 에이전트가 무엇을 할 수 있으며 어떻게 상호작용해야 하는지 명확히 이해하게 하는 일
  - 백엔드 에이전트 설계가 지원 기능을 결정
  - 실제로 그 능력이 발견 가능성, 직관성, 실사용성을 갖추는지는 사용자 경험이 좌우함

- 효과적인 에이전트 UX
  - 인터페이스 자체를 통해 능력을 선제적으로 커뮤니케이션해 이 문제를 해결함
    - EX) 많은 채팅 기반 에이전트 : 입력창 아래에 제안 액션 버튼을 제공해 '주문 추적', '요약 생성', '회의 노트 작성'
      - 일반적이거나 컨텍스트적으로 관련 있는 기능을 강조함
      - 이 버튼 : 시각적으로 행동을 유도해 사용자가 특정 명령을 기억하거나 가능한 범위를 추측하지 않아도 지원 워크플로로 안내함
  - 확장형 메뉴나 '능력 카드'로 제공 기능을 구조적으로 열거하는 방식
    - EX) 그래픽 기반 에이전트 인터페이스 : 사이드바를 데이터 조회, 분석, 요약, 워크플로 자동화 섹션으로 구성
    - EX) 사용자 입력에 기반해 시스템이 기능을 추천하는 동적 제안
      - 사용자가 예약을 입력하면 에이전트는 '예약-[이름]과 회의, '예약-회의실', '예약-여행'처럼 의도를 예측해 쉽게 실행
     
  - 개방형 텍스트 입력에 주로 의존하는 시스템에서
    - 에이전트가 대화 속에서 스스로 능력을 분명히 알릴 수 있어야 함
    - EX) 세션 시작 시 콘텐츠 생성, 데이터 분석, 문서 요약을 도와드릴 수 있다고 주도적 소개
      - 그 이외 기능 요청 시 단순 거절 대신 대안 제시
   
  - 능력을 드러내는 것도 중요하지만 한 번에 너무 많은 옵션을 제시해 사용자를 압도하지 않는 것이 똑같이 중요함
    - 효과적인 디자인 : 점진적 공개를 우선하여 초기에는 핵심 기능을 보여주고 사용자가 익숙해질수록 고급 기능을 드러냄
    - 컨텍스트 관련성 역시 핵심

  - 다양한 모달리티에 적용됨
    - 텍스트 기반 채팅 인터페이스 : 빠른 응답 버튼과 예시 프롬프트가 명료성을 높임
    - 그래픽 대시보드 : 능력 메뉴와 툴팁이 화면을 과밀화하지 않고 기능을 전달함
    - 음성 에이전트 : 인지 부하를 피하기 위해 한번에 소수의 고우선 옵션만 간결히 안내해야 함
    - 생성형 UI : 자연어와 동적으로 생성되는 시각 출력물을 결합해 사용 가능한 능력을 즉시 시각화하고 실행 가능하게 만들어야 함

### 3.4.1 신뢰도와 불확실성 커뮤니케이션
- 에이전트 : 종종 결정적 규칙이 아닌 통계 모델에 기반한 확률적 환경에서 작동함
  - 모든 응답이나 행동이 동일한 신뢰도를 갖는 것은 아님
  - 불확실성을 효과적으로 전달하는 일 : 사용자 신뢰를 쌓고 사용자가 정보에 근거한 결정을 내리도록 돕는 데 필수
- 신뢰도 방식 표현
  - 명시적 진술 : 정답일 확률이 90%입니다
  - 시각적 신호 : 아이콘, 색상 기반 경고 그래픽 인터페이스의 신뢰도 미터
  - 행동 조정 : 신뢰도가 낮을 때 단정적 권고 대신 제안을 제공
- 불확실성이 높은 상황에서 과도한 자신감을 보여서는 안됨
- 반대로 위험이 낮은 상호작용에서 지나치게 모호하게 말하면 망설이거나 신뢰할 수 없는 인상을 줄 수 있음
- 사용자 기대와 상호작용의 중요도에 맞게 응답을 구성해야 함
  - 중요한 컨텍스트에서는 투명성이 필수
  - 위험이 낮은 상황에서는 좀 더 캐주얼하게 신뢰도를 제시할 수 있음

### 3.4.2 사용자 지침과 입력 요청
- 모호하거나 애매하고 상충되는 사용자 입력을 완벽히 해석할 수는 없음
  - 위험한 추정을 하기보다 명확한 질문을 하거나 사용자 지침을 구할 시점을 알아야 함
  - 잠재적 오류를 협업의 기회로 전환함
- 효과적인 에이전트
  - 모호성에 직면했을 때 초점을 맞춘 유용한 질문을 하도록 설계됨
    - 예를 들어 사용자가 시카고행 티겟 예약해 줘
      - 편도일까요 왕복일까요? 선호하는 여행 날짜가 있으신가요?처럼 응답함
  - 질문 방식도 중요
    - 질문은 명확하고 공손하며 컨텍스트 인식적이어야 하며 로봇 같거나 반복적인 표현을 피해야 함
  - 명확화가 필요한지 투명하게 설명해야 함
  - 한 번에 너무 많은 질문을 피해야 함
 

### 3.4.3 우아한 실패
- 불완전한 데이터, 모호한 입력, 기술적 한계, 예상치 못한 엣지 케이스 등으로 요청을 이해하지 못하거나 작업을 완료하지 못하는 상황이 발생함
  - 실패를 다루는 방식은 성공만큼이나 중요함
- 우아한 실패의 핵심
  - 문제를 투명하게 인정하고 유용한 설명을 제공하며 실행 가능한 다음 단계를 제시하는 것
    - 잘못된 답이나 무의미한 응답 대신 안내함
  - 흔한 실패 지점을 예측해 사전에 폴백 메커니즘을 갖추어야 함
  - 다단계 작업에서는 실패 시 진행 상태 보존이 동일하게 중요함
    - 처음부터 다시 시작하게 하지 말고 문제가 해결되면 이어서 진행할 수 있또록 진행 상황을 유지해야 함
  - 또 다른 핵심 : 사과와 공감의 언어
    - 문제 발생 시 인간적이고 배려 있는 방식으로 실패를 인정해야 함
  - 명확한 해결 경로를 제공해야 함
    - 장애가 발생했을 때 사용자가 어떤 선택지가 있는지 항상 알 수 있어야 함
  - 가능하다면 실패로부터 학습해야 함
    - 실패 지점을 기록하고 반복 이슈를 분석해 개발 프로세스에 반영하면 유사 실패 빈도를 줄일 수 있음

## 3.5 상호작용 설계에서의 신뢰
- 투명성과 예측 가능성 : 에이전트와 사용자 사이의 신뢰를 구축, 유지하는 데 가장 강력한 도구
  - 투명성
    - 에이전트의 능력과 제약을 명확히 전달하는 데서 시작함
    - 에이전트가 자신의 행동을 설명하면 (추천에 이른 과정, 요청을 거절한 이유, 모호한 지침을 어떻게 해석했는지 등) 사용자는 그 추론에 대한 가시성을 얻음
    - 사용자에게 과할 정도로 불필요한 세부 정보를 드러내야 한다는 의미가 아님
      - 사용자는 에이전트의 추론 전 과정을 볼 필요가 없음
      - 행동을 신뢰할 만큼의 적절한 인사이트 정도면 충분함
  - 예측 가능성
    - 다양한 상황에서의 일관된 행동을 통해 투명성을 보완함
    - 과거 상호작용을 바탕으로 에이전트의 반응을 예측할 수 있어야 함
      - 기술적으로 옳더라도 변덕스럽거나 일관되지 않은 행동은 신뢰를 빠르게 훼손함

- 신뢰 구축의 두 가지 핵심 요소 : 에이전트 행동의 예측 가능성과 신뢰성
  - 예측 가능성과 신뢰성은 사용자의 신뢰를 받는 기존 조건
    - 신뢰성 : 출력의 일관성에서 출발함
  - 신뢰성의 또 다른 핵심 : 시스템 복원력
    - 에이전트는 오류에서 회복하고 중단 상황에서도 상태를 유지하며 연쇄적으로 중단되지 않도록 설계해야 함
  - 신뢰성 : 기대를 일관되게 설정하고 충족하는 일
     
- 엣지 케이스도 신중히 처리해야 함
  - 불완전한 데이터, 상충하는 지침, 모호한 입력을 만나면 명확한 추가 질문, 중립적 폴백 응답, 요청 전달 가운데 예측 가능한 방식으로 대응해야 함
 
- 에이전트가 지켜야 할 패턴
  - 능력을 명확히 전달하기
  - 모달리티를 신중히 결합하기
  - 컨텍스트를 신중히 유지하기
  - 오류를 우아하게 처리하기
  - 신뢰 구축하기
