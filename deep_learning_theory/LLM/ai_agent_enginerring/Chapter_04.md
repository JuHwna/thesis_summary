# CHAPTER 04. 도구
- 도구
  - AI 에이전트가 추가 정보와 컨텍스트를 검색하고 작업을 수행하며 환경과 의미 있게 상호작용하도록 함
  - 에이전트가 원하는 결과를 달성하기 위해 수행할 수 있는 특정 기능 또는 일련의 작업
  - 단일 단계 작업부터 고급 추론과 문제 해결 능력이 필요한 복잡한 다단계 작업까지 다양함
  - 정보를 검색해 제공하는 데 그치지 않고 실제 변화를 만들어 내는 에이전트를 원한다면 도구가 반드시 필요함
  - 독립적으로 개발, 테스트, 최적화할 수 있는 모듈형 구성 요소로 통합되어 복잡한 행동이 가능한 일관된 시스템을 이룸
- 도구의 설계와 구현
  - AI 에이전트의 전반적인 기능과 효과를 좌우함
 
## 4.1 랭체인 기초
- 랭체인의 중심에는 프롬프트를 처리하고 응답을 생성하는 파운데이션 모델을 감싼 채팅 모델(chat model)이 있음
- init_chat_model()에 모델 이름을 파라미터로 전달해 다양한 제공업체의 LLM을 동일한 채팅 모델 인터페이스로 초기화할 수 있음
  - 오픈AI, 앤트로픽, 아마존, 구글 등 다양한 제공업체의 모델을 사용할 수 있음
  - 각 제공업체별로 대응되는 랭체인 통합 패키지 설치한 뒤 init_chat_model()로 초기화함
    - from langchain_openai import ChatOpenAI => 그록은 같음
    - from langchain_anthropic import ChatAnthropic
    - from langchain_google_genai import ChatGoogleGenerativeAI
- 대화 컨텍스트를 유지하기 위해 상호작용을 메시지로 구조화함
  - 주요 메시지 유형
    - HumanMessage : 사용자 입력을 나타냄
    - AIMessage : 모델의 응답을 나타냄
    - ToolMessage : 도구 호출 결과를 표현함

- 도구 : 모델이 텍스트 생성 능력을 넘어 기능을 확장하도록 호출할 수 있는 외부 함수
  - ex) API 호출, 데이터베이스 조회, 계산 수행 등
  - 랭체인 : @tool 데코레이터를 사용해 도구를 정의함
    - 함수를 등록하고 입력과 출력 스키마를 자동으로 생성함
      - from langchain_core.tools import tool
  - 도구를 정의한 뒤에 .bind_tools()로 모델에 바인딩함
    - 모델이 사용자 입력에 응답하는 과정에서 해당 도구를 선택해 호출할 수 있음
  - 모델과 상호작용할 때 현재 대화를 나타내는 메시지 목록을 .invoke()에 전달함
    - 모델이 도구 호출을 결정하면 도구 호출이 출력에 포함됨
- 랭체인 기반 시스템의 토대
  - 구성 요소 : 채팅 모델, 메시지, 도구, 도구 호출
 

### 4.1.1 로컬 도구
- 로컬 도구
  - 로컬에서 실행하는 도구
  - 특정 작업에 맞춘 사전 정의 규칙과 로직을 따라 만듦
  - 만들고 수정하기 쉽고 에이전트와 함께 배포함
  - 기존 프로그래밍이 더 잘하는 영역에서 언어 모델의 약점을 보완할 수 있음
  - 정확성, 예측 가능성, 단순성을 제공함
  - 로직이 명시적으로 정의됨
    - 대체로 예측 가능하고 신뢰할 수 있음
- 메타데이터(도구의 이름, 설명, 스키마) 중요함
  - 모델은 해당 메타데이터를 바탕으로 어떤 도구를 호출할지 결정함
  - 중요한 과정
    - 범위를 좁게 답고 정확한 이름을 선택함 => 이름이 일반적이면 필요하지 않은 상황에도 LLM이 도구를 호출할 수 있음
    - 명확하고 구별되는 설명을 작성함 => 여러 도구에 걸쳐 지나치게 넓거나 겹치는 설명은 혼란과 성능 저하를 초래함
    - 엄격한 입력/출력 스키마를 정의함 => 명시적 스키마는 파운데이션 모델이 도구를 언제, 어떻게 사용할지 정확히 이해하도록 도와 오작동을 줄임
   
- 로컬 도구 단점
  - 확장성
    - 로컬 도구의 설계, 구축, 배포는 번거롭고 시간과 노력이 많이 들음
    - 사용 사례 전반에 걸쳐 공유하기가 더 어려움
    - 도구를 라이브러리로 노출해 여러 에이전트 사용 사례에서 공유할 수 있지만 실제로는 대규모에서 어려움
  - 중복
    - 로컬 도구를 사용하려는 모든 팀이나 에이전트 배포는 해당 에이전트 서비스와 함께 같은 라이브러리를 배포해야 함

    - 이 도구들에 변경 사항을 배포하려면 이를 사용하는 각 에이전트 서비스의 배포를 조율해야 함

  - 유지보수
    - 환경이나 요구사항이 바뀌면 수작업으로 만든 도구는 업데이트와 조정이 자주 필요할 수 있음
    - 지속적 유지보수 : 리소스를 많이 소모, 에이전트 서비스의 재배포가 필요함

- 수작업으로 만든 도구
  - 파운데이션 모델이 가진 약점을 해결하는데 유용함
  - EX) 수학 연산, 단위 변환, 계산기 연산, 캘린더 변경, 날짜와 시간 연산, 지도와 그래프 연산 등

### 4.1.2 API 기반 도구
- API 기반 도구(API-Based Tools)
  - 자율 에이전트가 외부 서비스와 상호작용하도록 하여 로컬에서 수행하기 어려운 데이터 처리나 액션 실행을 가능하게 기능을 확장함
  - 애플리케이션 프로그래밍 인터페이스(API)를 활용해 공용 또는 사설 서비스와 통신하며 에이전트의 기능을 동적으로 확장하고 확장성을 높이는 수단을 제공함
  - 에이전트가 여러 외부 시스템과 통합하거나 실시간 데이터를 가져오거나 내부에서 처리하기엔 자원 소모가 큰 복잡한 연산을 수행해야 하는 상황에서 특히 유용함(실시간 데이터, 고부하 연산, 외부 액션 수행 능력을 부여해 그 자체로는 수행할 수 없는 작업을 가능하게 만듦)
  - EX) 날씨 정보, 주식 시장 데이터, 번역 서비스 등
  - 에이전트가 수행할 수 있는 작업의 범위를 크게 넓어짐
  - 모델을 재학습할 필요 없이 다양한 외부 서비스를 통합해 에이전트의 기능을 넓힘
  - 또다른 장점 : 실시간 데이터 접근 기능
    - 외부 소스로부터 최신 정보를 받아와 항상 최신 데이터에 기반한 응답과 액션을 제공함

- API 상호작용 정의, 응답 처리, 도구를 에이전트 워크플로에 통합하는 단계가 필요함
  - 이 과정으로 에이전트는 외부 데이터 소스를 원활하게 통합할 수 있음
  - 전체적인 기능과 효율성을 높일 수 있음

- API 도구를 설계할 때 중점적으로 두어야할 사항
  - 신뢰성, 보안성 우아한 실패(Graceful Failure)
  - 외부 서비스는 장애가 발생할 수 있으므로 폴백 메커니즘이나 명확한 오류 메시지를 제공해야 함
  - API의 요청 제한을 주의해 서비스 중단을 방지하고 개인정보 보호법을 준수하기 위해 사용자 데이터를 익명화하거나 마스킹해야 함


### 4.1.3 플러그인 도구
- 플러그인 도구(Plug-in Tools)
  - 모듈화되어 잇음
  - 최소한의 커스터마이징만으로 AI에이전트 프레임워크에 통합할 수 있는 구성 요소
  - 기존 라이브러리, API, 서드파티 서비스를 활용해 광범위한 개발 작업 없이도 에이전트의 기능을 확장함
  - 에이전트의 기능을 빠르게 배포하고 확장할 수 있게 해줌
    - 사전에 설계된 모듈로서 적은 노력으로 AI 시스템에 통합할 수 잇음
  - 주요 LLM 플랫폼과 오픈소스 커뮤니티에서 플로그인 도구 통합 기능을 표준 기능으로 제공함
  - 대규모 맞춤 개발 없이도 AI 에이전트의 역량을 확장하는 강력한 수단
