# CHAPTER 04. 도구
- 도구
  - AI 에이전트가 추가 정보와 컨텍스트를 검색하고 작업을 수행하며 환경과 의미 있게 상호작용하도록 함
  - 에이전트가 원하는 결과를 달성하기 위해 수행할 수 있는 특정 기능 또는 일련의 작업
  - 단일 단계 작업부터 고급 추론과 문제 해결 능력이 필요한 복잡한 다단계 작업까지 다양함
  - 정보를 검색해 제공하는 데 그치지 않고 실제 변화를 만들어 내는 에이전트를 원한다면 도구가 반드시 필요함
  - 독립적으로 개발, 테스트, 최적화할 수 있는 모듈형 구성 요소로 통합되어 복잡한 행동이 가능한 일관된 시스템을 이룸
- 도구의 설계와 구현
  - AI 에이전트의 전반적인 기능과 효과를 좌우함
 
## 4.1 랭체인 기초
- 랭체인의 중심에는 프롬프트를 처리하고 응답을 생성하는 파운데이션 모델을 감싼 채팅 모델(chat model)이 있음
- init_chat_model()에 모델 이름을 파라미터로 전달해 다양한 제공업체의 LLM을 동일한 채팅 모델 인터페이스로 초기화할 수 있음
  - 오픈AI, 앤트로픽, 아마존, 구글 등 다양한 제공업체의 모델을 사용할 수 있음
  - 각 제공업체별로 대응되는 랭체인 통합 패키지 설치한 뒤 init_chat_model()로 초기화함
    - from langchain_openai import ChatOpenAI => 그록은 같음
    - from langchain_anthropic import ChatAnthropic
    - from langchain_google_genai import ChatGoogleGenerativeAI
- 대화 컨텍스트를 유지하기 위해 상호작용을 메시지로 구조화함
  - 주요 메시지 유형
    - HumanMessage : 사용자 입력을 나타냄
    - AIMessage : 모델의 응답을 나타냄
    - ToolMessage : 도구 호출 결과를 표현함

- 도구 : 모델이 텍스트 생성 능력을 넘어 기능을 확장하도록 호출할 수 있는 외부 함수
  - ex) API 호출, 데이터베이스 조회, 계산 수행 등
  - 랭체인 : @tool 데코레이터를 사용해 도구를 정의함
    - 함수를 등록하고 입력과 출력 스키마를 자동으로 생성함
      - from langchain_core.tools import tool
  - 도구를 정의한 뒤에 .bind_tools()로 모델에 바인딩함
    - 모델이 사용자 입력에 응답하는 과정에서 해당 도구를 선택해 호출할 수 있음
  - 모델과 상호작용할 때 현재 대화를 나타내는 메시지 목록을 .invoke()에 전달함
    - 모델이 도구 호출을 결정하면 도구 호출이 출력에 포함됨
- 랭체인 기반 시스템의 토대
  - 구성 요소 : 채팅 모델, 메시지, 도구, 도구 호출
 

### 4.1.1 로컬 도구
- 로컬 도구
  - 로컬에서 실행하는 도구
  - 특정 작업에 맞춘 사전 정의 규칙과 로직을 따라 만듦
  - 만들고 수정하기 쉽고 에이전트와 함께 배포함
  - 기존 프로그래밍이 더 잘하는 영역에서 언어 모델의 약점을 보완할 수 있음
  - 정확성, 예측 가능성, 단순성을 제공함
  - 로직이 명시적으로 정의됨
    - 대체로 예측 가능하고 신뢰할 수 있음
- 메타데이터(도구의 이름, 설명, 스키마) 중요함
  - 모델은 해당 메타데이터를 바탕으로 어떤 도구를 호출할지 결정함
  - 중요한 과정
    - 범위를 좁게 답고 정확한 이름을 선택함 => 이름이 일반적이면 필요하지 않은 상황에도 LLM이 도구를 호출할 수 있음
    - 명확하고 구별되는 설명을 작성함 => 여러 도구에 걸쳐 지나치게 넓거나 겹치는 설명은 혼란과 성능 저하를 초래함
    - 엄격한 입력/출력 스키마를 정의함 => 명시적 스키마는 파운데이션 모델이 도구를 언제, 어떻게 사용할지 정확히 이해하도록 도와 오작동을 줄임
   
- 로컬 도구 단점
  - 확장성
    - 로컬 도구의 설계, 구축, 배포는 번거롭고 시간과 노력이 많이 들음
    - 사용 사례 전반에 걸쳐 공유하기가 더 어려움
    - 도구를 라이브러리로 노출해 여러 에이전트 사용 사례에서 공유할 수 있지만 실제로는 대규모에서 어려움
  - 중복
    - 로컬 도구를 사용하려는 모든 팀이나 에이전트 배포는 해당 에이전트 서비스와 함께 같은 라이브러리를 배포해야 함

    - 이 도구들에 변경 사항을 배포하려면 이를 사용하는 각 에이전트 서비스의 배포를 조율해야 함
      - <img width="742" height="187" alt="image" src="https://github.com/user-attachments/assets/e3a235d5-7104-4987-be87-1506c7d9e1ef" />

  - 유지보수
    - 환경이나 요구사항이 바뀌면 수작업으로 만든 도구는 업데이트와 조정이 자주 필요할 수 있음
    - 지속적 유지보수 : 리소스를 많이 소모, 에이전트 서비스의 재배포가 필요함

- 수작업으로 만든 도구
  - 파운데이션 모델이 가진 약점을 해결하는데 유용함
  - EX) 수학 연산, 단위 변환, 계산기 연산, 캘린더 변경, 날짜와 시간 연산, 지도와 그래프 연산 등

### 4.1.2 API 기반 도구
- API 기반 도구(API-Based Tools)
  - 자율 에이전트가 외부 서비스와 상호작용하도록 하여 로컬에서 수행하기 어려운 데이터 처리나 액션 실행을 가능하게 기능을 확장함
  - 애플리케이션 프로그래밍 인터페이스(API)를 활용해 공용 또는 사설 서비스와 통신하며 에이전트의 기능을 동적으로 확장하고 확장성을 높이는 수단을 제공함
  - 에이전트가 여러 외부 시스템과 통합하거나 실시간 데이터를 가져오거나 내부에서 처리하기엔 자원 소모가 큰 복잡한 연산을 수행해야 하는 상황에서 특히 유용함(실시간 데이터, 고부하 연산, 외부 액션 수행 능력을 부여해 그 자체로는 수행할 수 없는 작업을 가능하게 만듦)
  - EX) 날씨 정보, 주식 시장 데이터, 번역 서비스 등
  - 에이전트가 수행할 수 있는 작업의 범위를 크게 넓어짐
  - 모델을 재학습할 필요 없이 다양한 외부 서비스를 통합해 에이전트의 기능을 넓힘
  - 또다른 장점 : 실시간 데이터 접근 기능
    - 외부 소스로부터 최신 정보를 받아와 항상 최신 데이터에 기반한 응답과 액션을 제공함

- API 상호작용 정의, 응답 처리, 도구를 에이전트 워크플로에 통합하는 단계가 필요함
  - 이 과정으로 에이전트는 외부 데이터 소스를 원활하게 통합할 수 있음
  - 전체적인 기능과 효율성을 높일 수 있음

- API 도구를 설계할 때 중점적으로 두어야할 사항
  - 신뢰성, 보안성 우아한 실패(Graceful Failure)
  - 외부 서비스는 장애가 발생할 수 있으므로 폴백 메커니즘이나 명확한 오류 메시지를 제공해야 함
  - API의 요청 제한을 주의해 서비스 중단을 방지하고 개인정보 보호법을 준수하기 위해 사용자 데이터를 익명화하거나 마스킹해야 함


### 4.1.3 플러그인 도구
- 플러그인 도구(Plug-in Tools)
  - 모듈화되어 잇음
  - 최소한의 커스터마이징만으로 AI에이전트 프레임워크에 통합할 수 있는 구성 요소
  - 기존 라이브러리, API, 서드파티 서비스를 활용해 광범위한 개발 작업 없이도 에이전트의 기능을 확장함
  - 에이전트의 기능을 빠르게 배포하고 확장할 수 있게 해줌
    - 사전에 설계된 모듈로서 적은 노력으로 AI 시스템에 통합할 수 잇음
  - 주요 LLM 플랫폼과 오픈소스 커뮤니티에서 플로그인 도구 통합 기능을 표준 기능으로 제공함
  - 대규모 맞춤 개발 없이도 AI 에이전트의 역량을 확장하는 강력한 수단

- 오픈 AI의 플러그인 생태계
  - 실시간 웹 검색부터 전문 코드 생성기까지 강력한 확장을 제공함
  - 챗GPT 제품 내에서만 사용 가능, 공용 API에서는 지원되지 않음
  - EX) 익스피디아나 재피어 같은 1차 제공 플러그인
    - 오픈 AI의 Completions이나 Chat 엔드포인트에서 호출할 수 없음
    - 유사한 과정을 직접 구현하려면 랭체인 같은 프레임워크를 사용해 플러그인 기능을 모사하는 커스텀 함수 호출 계층을 구축해야 함
- 앤트로픽의 클로드
  - 도구 사용 기능 전체를 Message API(또는 아마존 베드락, 구글 클라우드 버텍스 ai 같은 플랫폼)를 통해 직접 제공함
  - 사용자는 커스텀 툴을 등록하거나 앤트로픽이 제공하는 도구를 사용하면 됨 => 클로드는 추론 시점에 이를 호출할 수 있음
  - 별도의 UI 없이 API 중심 접근 방식을 취하기 때문에 콘텐츠 모더레이션, 편향 탐지, 도메인 특화 서비스 등을 클로드 기반 워크플로에 쉽게 통합할 수 잇음
- 구글의 제미나이
  - Vertex AI API를 통해 함수 호출을 지원함
  - 개발자는 FunctionCallingConfig 내에서 사용할 함수를 선언하고 제미나이가 구조화된 호출로 이를 실행하도록 설정할 수 있음
  - 자연어 이해, 이미지 인식 등 어떤 기능이든 사전에 정의한 함수를 호출하고 반환된 인수를 코드에서 처리하면 됨
    - 애플리케이션과 모델 사이에 별도의 전용 UI 계층이 필요 없다는 점에서 유연함
   
- 마이크로소프트의 파이 모델
  - Microsoft Foundry에서 제공
  - 애저의 AI 검색, 문서 처리, 데이터 시각화 API 등과 같은 다른 서비스와 자연스럽게 통합됨
  - 애저의 생산성 및 분석 툴과 긴밀히 연결되어 있음
  - 모델을 호출하면 구조화된 출력을 받아 기존 애저 워크플로에 바로 연결할 수 있어 별도의 컨텍스트 전환이 필요 없음
 

- 플러그인 도구의 한 가지 장점
  - 모델 실행 계층에서의 통합 가능성
    - 이러한 도구를 기존 워크플로에 거의 영향을 주지 않고 ai 모델에 추가할 수 있음

- 한계점
  - 강력하지만 로컬 또는 원격에서 제공되는 맞춤형 툴처럼 높은 수준의 커스터마이징과 적응성을 제공하지는 않음
  - 다양한 작업을 처리할 수 있도록 범용적으로 설계되었지만 모든 애플리케이션의 세부 요구사항에 맞추어 최적화되어 있지는 않음

- 오픈소스 커뮤니티
  - 플러그인 도구의 개발에 활발히 기여하며 협력적이고 혁신적인 환경을 조성함
  - ex) 허깅페이스의 Transformers 라이브러리 : 자연어 처리 작업을 위한 다양한 사전 학습 모델과 플러그인 도구를 제공함
  - ex) Glama.ai, mcp.so 같은 플랫폼 : 수많은 MCP 서버를 통합해 간단한 유틸리티부터 복잡한 상태 기반 서비스까지 탐색하고 활용할 수 있도록 지원함

- 플러그인 도구의 미래가 매우 밝음
  - 지속적인 기술 발전과 산업 전반의 확산으로 ai 개발에서의 중요성이 더욱 커지고 있음
  - 플러그인 도구의 상호운용성과 표준화 : 중요해질 예정
    - 공통 표준과 프로토콜을 수립하면 다양한 AI 플랫폼과 시스템 간의 통합이 쉬워짐
    - 개발자는 여러 출처의 플러그인 도구를 유연하게 활용할 수 있음
   
### 4.1.4 MCP
- 모델 컨텍스트 프로토콜(MCP)
  - 앤트로픽이 처음 제안한 표준, 이후 오픈AI, 구글 딥마인드, 마이크로소프트 등 주요 기업이 채택함
  - 모델에 독립적인 통합 표준을 제공해 LLM과 외부 시스템을 연결하는 통일된 방식의 인터페이스 역할을 함
  - 모든 데이터 소스나 툴이 하나의 명확한 인터페이스를 통해 연결될 수 있게 함
  - 두 가지 주요 구성 요소
    - MCP 서버
      - 표준화된 JSON-RPC 2.0 인터페이스를 통해 데이터나 서비스를 노출하는 웹 서버
      - 클라우드 오브젝트 스토리지, SQL 데이터베이스, 기업용 고객 관계관리시스템(CRM), 내부 비즈니스 로직 등 MCP 사양을 구현하기만 하면 어떤 시스템이든 서버로 래핑할 수 있음
    - MCP 클라이언트
      - MCP를 사용하는 에이전트 또는 LLM 애플리케이션
      - 클라이언트는 JSON-RPC 요청을 전송하고 구조화된 JSON 응답을 받음
      - 프로토콜이 일관되므로 개발자는 서버의 내부 작동 원리를 몰라도 노출된 메서드만 알면 됨

- MCP는 내부적으로 HTTPS 도는 Websocker에서 JSON-RPC 2.0을 사용함
  - 서버는 자신이 제공하는 메서드(EX : listFiles, getRecord, runAnalysis)와 입출력 스키마를 공개하고 클라이언트는 이 '메서드 카탈로그'를 조회해 어떤 메서드를 어떤 파라미터로 호출할지 추론함
  - 도구 호출이 결정되면 MCP 클라이언트는 해당 호출을 JSON-RPC 페이로드로 래핑해 서버로 전송하고 응답을 기다림
  - 양쪽이 동일한 프로토콜을 사용하므로 플랫폼 간 상호운용성을 쉽게 구축할 수 있음
 

- MCP 문제
  - 여러 보안 이슈가 제기됨
    - 여러 에이전트가 MCP 엔드포인트를 공유하는 경우 인증, 접근 제어, 잠재적인 공격 벡터와 관련된 문제
    - 문제해결을 위해 추가적인 네트워크 정책이나 프록시 레이어에 의존함
      - 핵심 MCP 스펙은 아직 단일한 표준 보안 솔루션을 의무화하지 않았음
- MCP는 여러 에이전트에 걸친 도구 재사용이라는 중요한 문제를 해결함
  - 한 번 서비스를 MCP를 통해 노출하면 여러 에이전트가 각 에이전트마다 커스텀 어댑터를 다시 작성하지 않고도 그 메서드를 발견하고 호출할 수 있음
  - 모듈식으로 재사용 가능한 아키텍처를 구성해 에이전트 개발을 더 쉽게 만듦

- MCP는 대규모로 AI 에이전트를 설계, 배포, 운영하는 방식에 의미 있는 진전을 이룩함
  - 메서드를 노출하고 소비하기 위한 단일의 표준화된 JSON-RPC 인터페이스를 정의함
  - 서비스 구현을 에이전트 로직에서 분리함
    - 여러 에이전트가 별도의 커스텀 통합 없이 동일한 도구를 재사용할 수 있음
  - 다양한 서비스를 빠르고 신뢰성 있게 통합하는 것


### 4.1.5 상태 유지 도구
- 상태 유지 도구
   - 로컬 스크립트, 외부 API, MCP에서 배포된 서비스 모두에 존재할 수 있음
   - 같은 약점을 가지괴 있음
     - 파운데이션 모델에 지속적인 상태를 직접 조작할 권한을 부여하면 모델이 파괴적인 실수를 하거나 악성 사용자가 악용할 수 있다는 점
     - EX) 데이터베이스 성능을 최적화한다며 프로덕션 테이블의 절반 행을 삭제해 중요한 레코드를 지운 사건이 있음
- 위험을 줄이기 위해선 '임의 SQL 실행' 엔드 포인트를 노출하는 대신 벙뮈가 좁게 정의된 작업만 도구로 등록해야 함
  - EX) get_user_profile(user_id) 도구나 add_new_customer(record) 도구처럼 단일하고 검증된 쿼리나 프로시저를 캡슐화함
  - 읽기 전용 접근만 필요한 에이전트에게는 데이터를 삭제하거나 수정할 권한을 줘서는 안됨
 
- 사용 사례상 자유형 쿼리가 반드시 필요하다면
  - 철저한 데이터 정화(sanitization)와 접근 제어를 구현해야 함
  - 모든 도구 호출을 로깅해 이상 행위를 탐지하고 포렌식 분석을 지원하는 것 => 강력히 권장됨
- 궁극적으로 '최소 권한 원칙' : 설계의 기준이 되어야 함
  - 기능을 제한하고 입력을 정화하며 최소 권한을 강제하고 완전한 가시성을 유지해야 함


## 4.2 도구 개발 자동화
- 코드 생성
  - AI 에이전트가 자율적으로 코드를 작성해 소프트웨어 애플리케이션을 개발하고 유지보수하는 데 필요한 시간과 노력을 크게 줄이는 기술
  - 에이전트가 실시간으로 자신의 도구를 작성해 작업을 수행하거나 새로운 API와 상호작용할 때 잠재력이 두드러짐
  - 동적 접근 방식 => AI 에이전트가 기능을 스스로 확장하고 적응하도록 만들어 유연성과 문제 해결 능력 크게 강화하게 됨

### 4.2.1 파운데이션 모델을 활용한 도구 개발
- 직접 도구를 생성함
  - LLM에 API 명세나 샘플 입력값 제공 시 => 초기 래퍼(wrapper), 헬퍼 함수, 또는 상위 수준의 '원자적' 연산을 자동으로 생성할 수 있음
  - 모델이 임시로 코드를 작성해 안전한 샌드박스 환경에서 실행한 뒤 스스로 결과를 평가할 수 있음
- 이러한 접근 방식 : 복잡한 API 환경을 다룰 때 특히 유용함
  - API가 변경될 때마다 동일한 생성-정제(generate-and-refine) 루프를 다시 실행하면 도구를 최신 상태로 유지할 수 있음
 
- 명확한 검증 기준(테스트, 응답 검증, 스키마 강제)과 개발자의 감독이 필수적


### 4.2.2 실시간 코드 생성
- 실시간 코드 생성
  - AI 에이전트가 작동 중 필요에 따라 코드를 작성하고 실행하는 기능
  - 에이전트는 특정 작업을 수행하기 위해 새로운 도구를 생성하거나 기존 도구를 수정할 수 있음
    - 그 결과 매우 높은 적응력을 갖게 됨

- 에이전트가 현재 작업을 분석하고 이를 수행하기 위해 필요한 단계를 결정하는 것부터 시작ㄱ됨
  - 이해한 내용을 바탕으로 코드 스니펫을 작성하고 이를 실행함
  - 코드가 기대한 대로 작동하지 않으면 반복적으로 수정하며 원하는 결과에 도달할 때까지 개선함
  - 시행착오 기반의 반복적 과정 => 에이전트가 도구를 지속적으로 정제하고 성능을 향상시키며 기능을 자율적으로 확장할 수 있게 함
 
- 장점 : 적응성과 효율성 측면에서 강력한 이점 제공
  - 즉석에서 코드를 생성하는 능력을 통해 ai 에이전트는 새로운 작업과 환경에 신속하게 대응할 수 있음 => 적응력

- 단점 : 여러 가지 과제와 위험 존재
  - 가장 큰 문제 : 품질 관리
    - 자율적으로 생성된 코드의 품질과 보안을 보장하지 못하면 시스템 오류, 보안 침해 등 심각한 문제가 발생할 수 있음
    - 견고한 보안 조치와 감독 체계를 반드시 마련해야 함
  - 재현성 문제
    - 에이전트가 매번 도구를 처음부터 다시 생성한다면 => 예측 가능성을 잃게 됨
    - 한 번의 호출에서 성공했다고 해서 다음 호출에서도 동일한 결과를 보장할 수 없음
    - 에이전트의 일관된 작동을 보증하기 어렵게 됨
  - 리소스 소비
    - 실시간 코드 생성과 실행은 많은 연산 자원과 메모리를 요구함
    - 시스템 성능의 여러 측면에 걸쳐 가드레일을 설정해 위험 완화 필요
   
## 4.3 도구 사용 설정
- 파운데이션 모델 API는 파라미터 tool-choice를 통해 모델의 도구 사용 방식을 명시적으로 제어할 수 있음
  - 파운데이션 모델 중심의 유연한 호출 방식에서 결정적 기능으로 전환할 수 있음
  - 'auto' 모드 : 모델이 컨텍스트에 따라 도구 호출 여부를 스스로 결정=> 일반적인 용도에 적합함
  - 'any' 또는 'required' 모드 : 모델이 최소 한 개의 도구를 반드시 호출하도록 강제 => 도구의 출력이 필수적인 경우에 이상적
  - 'none' 모드 : 모든 도구 호출이 차단되며 이는 제어된 출력이나 테스트 환경에서 활용할 수 있음
 
- 신뢰할 수 있는 폴백 메커니즘과 후처리 체계를 반드시 마련해야 함
  - 에이전트가 도구 호출을 건너뛰거나 잘못된 JSON을 생성하거나 오류가 발생한 도구를 실행하는 등 예상치 못한 오작동을 보일 수 있기 때문에
  - 모델의 각 응답 후에 다음 사항을 점검해야 함
    - 모델이 올바른 도구를 호출했는지
    - 유효한 JSON을 생성했는지
    - 런타임 오류 없이 성공적으로 실행되었는지
  - 문제가 발견되면 다음과 같은 교정 절차 적용해야 함
    - 스키마 검증
      - jsonschema나 Pydantic을 사용해 누락된 필드나 잘못된 구조를 탐지함
      - 도구 호출이 누락된 경우 자동으로 실행하고 json이 유효하지 않다면 모델에 수정 프롬프트를 보냄
    - 지능적 재시도
      - 일시적 오류에는 지수 백오프 같은 구조화된 재시도 로직을 적용하거나 전체 과정을 다시 시작하지 않고 문제된 부분만 재생성함
    - 안정적인 폴백
      - 재시도가 실패하면 백업 모델이나 서비스를 사용하거나 사용자에게 추가 입력을 요청하거나 캐시된 데이터를 활용하거나 안전한 기본값을 반환함
    - 로깅
      - 프롬프트, 도구 호출, 검증 오류, 재시도, 폴백 등 모든 단계를 기록해 관측 가능성을 확보하고 디버깅과 개선에 활용함
