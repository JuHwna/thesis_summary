# Chapter 08. 데이터셋 엔지니어링
- 모델의 품질 : 학습 데이터의 품질에 달려 있음.
- 데이터셋 엔지니어링의 목표 : 최고의 모델을 학습할 수 있는 데이터셋을 만드는 것
  - 할당된 예산 내에서 이를 달성하는 것
 
- 데이터 운영
  - 더 이상 여유 시간에 처리하는 부수적인 업무가 아니라 그 자체로 전문적인 전담 역할이 되었음
 
- 데이터 환경에 대한 개요와 자체 데이터셋을 구축할 때 고려해야할 사항들
- 데이터 큐레이션부터 시작해서 어떤 데이터가 필요한지, 얼마나 많이 필요한지, 고품질 데이터란 무엇인지 같은 질문을 다룸
  - 데이터 합성과 처리 기법들을 살펴봄
- 데이터 큐레이션, 생성, 처리는 순차적인 과저을 따르지 않고 여러 단계를 오가며 작업하게 됨

- 같은 모델이라도 학습 단계마다 학습하려는 능력이 다르기 때문에 각각 다른 속성을 가진 데이터셋이 필요함
  - ex) 사전 학습에서는 데이터 양을 보통 토큰 수로 측정
  - 지도 파인튜닝에서는 예시 수로 측정하는 등 세부적인 접근 방식에는 차이가 있음
 

- 데이터 중심 AI의 관점

## 8.1 데이터 큐레이션
- 데이터 큐레이션 : 모델이 어떻게 학습하는지, 학습에 도움이 되는 자원이 무엇인지 이해하는 분야
- 어떤 데이터가 필요한지는 하려는 일과 모델에게 무엇을 가르치고 싶은지에 달려 있음.
  - 자기 지도 학습 파인튜닝 : 데이터 시퀀스가 필요
  - 지시 파인튜닝 : (지시) 응답 형식의 데이터 필요
  - 선호도 파인튜닝 : (지시, 선호 응답, 비선호 응답) 형식 필요
  - 보상 모델 학습 : 선호도 파인튜닝과 같은 형식을 쓰거나 ((지시,응답), 점수) 형식으로 각 예시에 주석이 달린 점수가 있는 데이터를 사용할 수 있음


- 학습 데이터는 모델이 학습했으면 하는 행동을 먼저 보여줘야 함
  - 생각의 사슬 (CoT)
    - 모델이 단계별 응답을 생성하도록 가르치려면 학습 데이터에 CoT 응답이 들어 있어야 함
   
  - 도구 사용
    - 도구 사용 예시를 다시 보여주면 모델의 도구 사용 능력을 더 확실히 키울 수 있음
    - 도구 사용 데이터를 만들 때는 보통 도메인 전문가를 활용함
    - 시뮬레이션이나 다른 합성 기법을 활용해 도구 사용 데이터를 생성함

- 대화 인터페이스가 있는 애플리케이션용 데이터를 큐레이션할 때는 싱글 턴 데이터가 필요한지, 멀티 턴 데이터가 필요한지, 아니면 둘 다 필요한지 고려해야 함
  - 싱글 턴 데이터 : 모델이 개별 지시에 답하는 법을 가르침
    - 간단해서 구하기 쉬움
  - 멀티 턴 데이터 : 모델에게 작업을 해결하는 방법을 가르침
    - 특별히 만든 시나리오나 더 복잡한 상호작용이 필요한 경우가 많음

- 데이터 큐레이션은 모델이 새로운 행동을 배우도록 돕는 새 데이터를 만드는 것뿐만 아니라 나쁜 행동을 잊게 하려고 기존 데이터를 없애는 것도 포함함

- 데이터 큐레이션의 세 가지 기준
  - 데이터 품질
    - 재료의 품질
  - 데이터 커버리지
    - 적절한 재료 조합을 갖는 것
  - 데이터 양
    - 재료를 얼마나 준비해야 하는지
   
### Chapter 8.1.1 데이터 품질
- 적은 양의 고품질 데이터가 많은 양의 노이즈가 있는 데이터보다 좋은 성능을 낼 수 있음
  - 노이즈가 있는 데이터란 관련이 없거나 일관성이 없는 데이터를 말함
  - Yi 모델 패밀리 연구팀
    - 신중하게 만든 1만 개의 지시가 수십만 개의 노이즈가 잇는 지시보다 훨씬 낫다
  - LIMA 2023 연구
    - 신중하게 큐레이션된 1000개의 프롬프트와 응답으로 파인튜닝한 650억 파라미터 라마 모델이 사람 주석자 기준으로 43%의 경우에서 GPT-4와 비슷하거나 더 좋은 답을 만든다는 것을 보여줌

- 고품질의 데이터란?
  - 고품질 : 데이터가 자신의 일을 효율적이고 안정적으로 하는 데 도움이 되는 데이터
  - 6가지 특성
    - 관련성 : 학습 예시가 모델이 학습하려는 작업과 관련이 있어야 함
    - 작업 요구사항 부합 : 주석은 작업의 요구사항에 부합해야 함
      - 정확한 또는 올바른 대신 부합이라는 표현을 쓴 이유 : 과제에 따라서는 정확하거나 올바른 응답이 사용자가 원하는 것이 아닐 수도 있기 때문
    - 일관성 : 주석은 예시들끼리, 그리고 주석자들 간에 일관되어야 함
      - 좋은 주석 가이드라인이 꼭 필요함
    - 올바른 형식 : 모든 예시는 모델이 기대하는 형식을 따라야 함
      - 불필요한 형식 토큰은 모델 학습을 방해할 수 있으므로 없애야 함
    - 충분한 고유성 : 데이터에서 고유한 예시를 말함
      - 모델 학습에서 중복은 편향을 만들고 데이터 오염을 일으킬 수 있음
    - 규정 준수 : 데이터는 모든 관련 내부 및 외부 정책(법률과 규정 포함)을 지켜야 함

### Chapter 8.1.2 데이터 커버리지
- 모델의 학습 데이터: 모델이 풀어야 할 문제들의 범위를 포괄해야 함
  - 애플리케이션의 다양한 사용 패턴을 담은 데이터를 확보하는 것 : 모델이 좋은 성능을 내는 핵심
- 데이터 다양성 : 좋은 커버리지를 확보하려면 데이터 다양성이 필수적

- 챗봇 같은 범용 용도인 경우 파인튜닝 데이터가 다양해야 하고 광범위한 주제와 말하기 패턴을 담아야 함
  - 딩 등의 연구(2023) : 채팅 언어 모델의 성능을 더 끌어올리는 가장 직접적인 방법 => 학습 과정에서 사용되는 데이터의 품질과 다양성을 늘리는 것
  - 네모트론 (2024)을 개발하기 위해 엔비디아 연구팀 : 과제 다양성, 주제 다양성, 지시 다양성을 가진 데이터셋을 만드는 데 집중
  - <The Data Addition Dilemma> (2024) : 경우에 따라 이질적인 데이터를 추가하면 오히려 성능이 나빠질 수 있음
  - 라마 논문
    - 성능 향상 : 주로 데이터 품질과 다양성 개선, 그리고 늘어난 학습 규모에 의해 이루어짐

- 올바른 데이터 조합은 어떻게 결정?
  - 간단한 방법: 실제 애플리케이션 사용 패턴을 맞춰 데이터 조합을 선택하는 것

### Chapter 8.1.3 데이터 양
- note
  - 사전 학습된 모델 위에 파인튜닝하는 것이 보통은 처음부터 학습시키는 것보다 효율적임
  - but, 파인튜닝이 오히려 더 나쁠 수 있는 상황들도 있음
    - 학습 데이터가 많을 때가 그럼
    - 경화 현상 : 사전 학습이 모델 가중치를 경화시켜서(얼려서) 파인튜닝 데이터에 잘 적응하지 못하게 만들 수 있음
      - 작은 모델일수록 큰 모델보다 이런 경화 현상에 더 취약함
     
- 데이터의 양을 결정하는 세 가지 요소
  - 파인튜닝 기법
    - 전체 파인튜닝 : 최고 성능을 낼 수 있지만 많은 데이터 필요 => (지시, 응답) 쌍이 수만 개에서 수백만개 있어야 함
    - LoRA 같은 PEFT 방법 : 수백 개나 수천 개 정도
   
  - 과제 복잡성
    - 간단한 과제 : 적은 데이터로 충분
  - 기본 모델의 성능
    - 기본 모델이 원하는 성능에 가까울수록 목표에 도달하는 데 필요한 예시가 적음

  - 위의 접근법 예시
    - 자기 지도 학습 -> 지도 학습

      - 먼저 법률 문서로 자기 지도 학습 방식으로 모델을 파인튜닝한 다음, (질의, 응답) 쌍으로 추가 파인튜닝할 수 있음
    - 관련성 낮은 데이터 -> 관련성 높은 데이터
      - 제품 리뷰의 감정을 분류하는 모델을 파인튜닝하고 싶지만 제품 감정 데이터는 적고 트윗 감정 데이터는 훨씬 많음
      - 먼저 트윗 감정 분류로 모델을 파인튜닝한 다음, 제품 감정 분류를 추가 파인튜닝할 수 있음
    - 합성 데이터 -> 실제 데이터

- 작은 데이터 셋으로 실험해보면 앞으로 데이터가 얼마나 더 필요한지 추정할 수 있음
  - 데이터셋이 커질수록 성능이 가파르게 올라가면 데이터를 늘렸을 때도 성능 좋아짐

### 8.1.4 데이터 수집과 주석
- 데이터 수집 목표 : 사용자 프라이버시를 존중하고 규정을 지키면서 필요한 품질과 다양성을 갖춘 충분한 크기의 데이터셋을 만드는 것
  - 공개 데이터 수집, 독점 데이터 구매, 데이터 주석 작업, 데이터 합성 등의 방법으로 데이터를 모으는 과정이 포함
 
- (지시, 응답) 데이터셋을 만드는 과정
  1. 원하는 특징을 가진 사용 가능한 데이터셋을 찾는다.
    - 10,000개의 예시로 이루어진 유망한 데이터셋 하나를 발견할 수 있음
  2. 낮은 품질의 지시를 제거함
    - 9000개의 예시가 남는다고 하자
  3. 낮은 품질의 응답을 가진 지시들을 따로 빼둔다
    - 그런 예시가 3000개 있다고 하자. 그러면 좋은 품질의 지시와 응답을 가진 6000개 예시가 남는다
  4. 좋은 품질의 지시 3000개에 대한 응답을 수동으로 작성함
    - 이제 데이터셋에는 총 9000개 고품질 예시를 갖게 됨
  5. 특정 주제 X에 대한 데이터가 부족하면, X에 관한 지시 템플릿 100개를 수동으로 만든다.
    - 이 템플릿 100개를 사용해 AI 모델로 지시 2000개를 합성함
  6. 이렇게 합성된 2000개 지시에 수동으로 주석을 추가함
    - 이제 데이터셋에는 총 11000개의 예시가 있게 됨

## 8.2. 데이터 증강 및 합성
- 데이터 생성을 자동화하는 것이 목표
  - 데이터 증강 : 기존 데이터(실제 데이터)에서 새로운 데이터를 만듦
    - 실제 데이터에서 나옴
  - 데이터 합성 : 실제 데이터의 특성을 모방하는 데이터를 생성함
    - 실제 데이터가 아님


### 8.2.1 데이터 합성을 하는 이유
- 데이터를 합성해서 데이터의 세 가지 중요한 측면인 양, 커버리지, 품질 개선할 수 있음
- 프라이버시 문제를 해결하고 모델을 증류하기 위해서도 데이터를 합성할 수 있음
  - 데이터 양 늘리기
    - 대규모로 데이터를 만들 수 있어서 AI 모델 학습과 테스트에 풍부한 데이터를 공급할 수 있다는 점
    - 이론적으로는 데이터가 많을수록 모델이 더 다양한 과제에 잘 적응할 수 있음
    - 실제 데이터를 구하기 어렵거나 부족한 경우에 유용함
  - 데이터 커버리지 늘리기
    - 특정 특성을 가진 데이터를 생성해서 모델 성능을 개선하거나 모델이 특정 행동을 하도록 만들 수 있음
    - 기존 데이터가 부족한 부분을 채우는 맞춤형 데이터를 만드는 것
  - 데이터 품질 향상
    - 보통은 사람이 만든 데이터가 품질이 좋음
    - BUT 때로는 사람의 근본적인 한계 때문에 사람이 만든 데이터가 AI가 만든 데이터보다 품질이 낮을 수 있음
   
  - 프라이버시 문제 해결
  - 모델 증류
    - 다른 모델의 행동을 모방하는 모델을 학습시키고 싶을 수 있음
    - 목표 : 보통 원래 모델과 비슷한 성능을 내면서도 더 저렴하거나 빠른 모델(증류된 모델)을 만드는 것
    - 원래 모델이 생성한 데이터로 증류된 모델을 학습시키는 방식으로 이뤄짐.

### 8.2.2 전통적인 데이터 생성 기법
- 절차적 생성 : 알고리즘으로 데이터를 생성하는 것
  - 게임에서 레벨, 맵, 아이템, 캐릭터 같은 콘텐츠를 실시간으로 만드는데 많이 쓰임
  - AI에도 적용할 수 있음
 
- 전통적으로 데이터 증강과 합성 : 규칙 기반 방식과 시뮬레이션 방식

- 규칙 기반
  - 미리 정해둔 규칙과 템플릿을 사용하는 것
  - 기존 데이터에 간단한 변형을 가해서 새로운 데이터를 절차적으로 생성할 수도 있음
    - 회전, 자르기, 크기 변경, 일부 지우기
  - 텍스트라면 문장의 뜻이나 감정이 바뀌지 않는 선에서 단어를 비슷한 단어로 바꿀 수 있음
  - 데이터의 편햐을 줄이는 데도 사용할 수 있음
  - 흥미로운 변형 중 하나 : 섭동
    - 기존 데이터에 노이즈를 넣어서 새로운 데이터를 생성하는 것
      - 데이터 샘플을 조금만 건드려도 모델을 속여서 잘못 분류하게 만들 수 있음
      - 문제 : 공격자가 ai 모델을 속여서 자신을 권한이 있는 직원으로 잘못 인식하게 하거나 자율주행차가 중앙 분리대를 차선으로 착각하게 해서 사고를 낼 수 있음
    - 섭통된 데이터로 모델을 학습시킬 수도 있음
      - 모델 성능을 향상시키고 공격에 더 견고하게 만들 수 있음

- 시뮬레이션
  - 가상의 시뮬레이션으로 실험을 통해 데이터를 모으는 것
    - 자율주행 시뮬레이션 엔진 : CARLA, 웨이모의 SimulationCity, 테슬라의 샌프란시스코시뮬레이션
  - Sim2Real : 시뮬레이션에서 학습한 알고리즘을 실제 세계에 적용하는 데 초점을 맞춘 분야
  - 모델에게 도구 사용법을 가르치는 데이터를 생성하는 데도 많이 쓰임
  - 현실에서 일어나기 힘든 사건의 데이터를 생성하는 데 유용함
 
### 8.2.3 AI 기반 데이터 합성
- 셀프플레이 : 게임 봇뿐만 아니라 일반적인 에이전트에도 유용함
- 역번역으로 번역 품질을 확인할 수 있음
- 사전 학습과 사후 학습 모두를 위한 데이터를 생성할 수 있음
  - 합성 데이터는 사전 학습보다 사후 학습에서 훨씬 더 자주 쓰임

- 사후 학습을 위한 데이터 합성이 더 흔한 이유
  - 지시 데이터와 선호도 데이터를 포함한 사후 학습 데이터가 보통 만들기 가장 힘들기 때문

#### 지시 데이터 합성
- 지시 파인튜닝 : 각 예시에 지시와 응답이 포함됨
  - AI로 지시를 만들 수도 있고 응답을 만들 수도 있으며 둘 다 만들 수도 있음
  - 지시 생성 : 활용 사롈르 충분히 다루는 지시를 만들려면 데이터셋에 포함하고 싶은 주제, 키워드, 지시 유형 목록으로 시작할 수 있음
    - 그 다음 목록의 항목마다 일정 수의 지시를 만듬
    - 템플릿 세트로 시작해서 템플릿당 일정 수의 예시를 만드는 방법도 있음
  - 응답 생성 : 지시 하나당 하나 또는 여러 개의 응답을 만들 수 있음

- 역지시 방법
  - 스토리, 책, 위키백과 글 같은 기존의 긴 고품질 콘텐츠를 가져와서 AI로 그런 콘텐츠를 유도할 수 있는 프롬프트를 생성하는 것
  - 응답에서 AI가 생성한 환각을 피하면서 더 고품질의 지시 데이터를 얻을 수 있음
 
#### 데이터 검증
- AI가 생성한 데이터의 품질 : 다른 AI 결과물을 평가하는 것과 같은 방식으로 측정할 수 있음
  - 기능적 정확성과 AI 평가자를 쓰는 것
 
- 기능적 정확성으로 검증할 수 없는 합성 데이터 : AI검증기를 사용하는 것이 일반적
  - AI 검증기 : 범용 AI 평가자거나 특화된 채점기일 수 있음

- 합성 데이터를 평가하는 기법이 많이 생겼지만 평가는 여전히 어려움
  - 다른 AI 애플리케이션과 마찬가지로 AI가 생성한 데이터의 최종 품질 테스트 : 실제 성능, 즉 모델 성능을 개선할 수 있는지 여부
    - 합성 데이터는 실제로 많은 모델의 성능을 개선하는 데 성공했음

#### AI 생성 데이터의 한계
- 품질츼 차이, 피상적 모방의 한계, 모델 성능 저하 가능성, 데이터의 계보가 불분명해지는 문제(AI가 데이터를 생성)

- 품질 관리
  - 데이터를 평가할 신뢰할 만한 방법과 지표를 개발할 수 있는 것 : 합성 데이터를 더 유용하게 만드는 데 꼭 필요함
 
- 피상적 모방
  - 모방 모델들이 교사 모델의 스타일을 모방하는 데는 뛰어나지만 사실적 정확성과 학습 데이터 범위를 벗어난 과제에 대한 일반화에서는 어려움을 겪을 수 있음
  - 더 심각한 문제 : 모방이 학생 모델로 하여금 환각을 일으키게 할 수 있다는 것
 
- 모델 성능 저하 가능성
  - AI가 생성한 데이터로 모델이 얼마나 학습할 수 있는지도 확실하지 않음
  - 모델 붕괴 : AI가 생성한 데이터를 반복적으로 학습에 사용하면 모델에 돌이킬 수 없는 결함이 생기고 시간이 지나면서 성능이 떨어짐
    - 사전 학습과 사후 학습 모두에서 일어날 수 있음

- 불분명한 데이터 계보
  - AI 모델은 학습 데이터의 영향을 받고 때로는 사용자도 모르게 그 내용을 그대로 출력할 수 있음
    - 위험 발생함
      - 모델 x를 사용해서 자신의 모델을 학습시킬 데이터를 만든다고 할 때
      - 모델 x가 저작권을 위반한 데이터로 학습됐다면 자신의 모델도 저작권을 위반할 수 있음

### 8.2.4 모델증류
- 모델 증류(지식 증류) : 작은 모델(학생)이 큰 모델(교사)을 모방하도록 학습시키는 방법
  - 큰 모델의 지식이 작은 모델로 증류되어 들어간다고 해서 증류라는 용어를 씀
  - ai가 만든 데이터를 단순히 보조 자료가 아닌 핵심적인 학습 데이터로 사용한다는 점에서 독특

- 전통적으로 모델 증류의 목표 : 배포용으로 더 작은 모델을 만드는 것
  - 학생 모델은 처음부터 학습시킬 수도 있고 알파카처럼 사전 학습된 모델에서 파인튜닝할 수도 있음


## 8.3 데이터 처리
- 데이터 처리 과정의 효율성을 높이는 데 도움이 되는 팁
  - 데이터 처리 단계는 시간과 컴퓨팅 자원을 절약할 수 잇는 순서라면 어던 순서로든 진행해도 됨
  - 모든 데이터에 스크립트를 적용하기 전에 처리 스크립트가 제대로 작동하는지 확인하기 위해 항상 테스트를 진행하자
  - 데이터를 원본에서 바로 수정하지 말자. 원본 데이터의 사본을 보관해야 하는 이유가 두 가지 있음
    - 이후 본인이나 다른 팀이 다른 애플리케이션에서 같은 데이터를 다르게 처리해야할 수 있음
    - 스크립트의 버그가 원본 데이터 자체를 망가뜨릴 수 있음
   
### 8.3.1 데이터 검사
