# Chapter 07. 파인튜닝
- 파인튜닝 : 모델 전체나 일부를 추가로 학습시켜 특정 작업에 맞게 모델을 조정하는 과정
  - 모델의 가중치를 조정하는 방식으로 모델을 변화시킴
  - 모델의 다양한 측면을 향상시킬 수 잇음
    - 도메인별 능력 향상+안전성 강화 + 모델의 지시 수행 능력 향상(가장 많이 사용되는 목적) => 특정 출력 스타일과 형식을 준수하는데 사용
  - 훨씬 더 많은 메모리를 필요로 함
    - 메모리 요구량 감소는 많은 파인튜닝 기법의 핵심 목표가 됨
    - 최근 파인튜닝 분야에서 메모리 효율성을 높인 기법 : **파라미터 효율적 파인튜닝(PEFT)**

## 7.1 파인튜닝 개요
- 파인튜닝 : 모든 능력을 갖춘 모델이 아닌, 필요한 기본적인 능력을 갖춘 기본 모델을 가지고 시작함
- 파인튜닝의 목표 : 모델이 특정 작업을 충분히 잘 수행하도록 만드는 것
- 파인튜닝의 한 방법 : 전이학습(Transfer learning)
  - 1976년 보지노프스키와 풀고시가 제안한 개념
  - 한 작업에서 얻은 지식을 새롭지만 관련된 작업에 활용해 학습 속도를 높이는 데 중점을 둠
  - 초기 대규모 성공 사례 : 구글의 다국어 번역 시스템
    - 포르투갈어-영어 및 영어- 스페인어 번역에 대한 지식을 전이해 학습 데이터에 포르투갈어-스페인어 예시가 없었음에도 포르투갈어를 스페인어로 변역할 수 잇었음

- 전이 학습
  - 딥러닝 초기부터 학습 데이터가 부족하거나 구하기 어려운 작업에 좋은 해결책
    - 풍부한 데이터가 있는 작업에서 기본 모델을 학습시킨 다음, 그 지식을 목표 작업에 활요하는 방식
  - LLM의 경우, 텍스트 완성(데이터가 풍부한 작업)에서 사전 학습으로 얻은 지식은 법률 질의 응답이나 Text-to-SQL 변환 같은 더 전문적인 작업(보통 데이터가 적은)으로 전이함
  - 표본 효율성을 높여 모델이 더 적은 예시로도 같은 행동을 학습할 수 있게 함
    - 표본 효율이 높은 모델은 적은 데이터로도 효과적으로 학습함
  - 이상적 : 모델이 학습해야 할 많은 부분이 이미 기본 모델에 내재되어 있고 파인튜닝은 단지 모델의 행동을 다듬는 과정

- 특정 기반 전이(feature-based trasfer)
  - 모델이 데이터에서 특성을 추출하도록 학습되며 주로 임베딩 벡터 형태로 추출된 특성을 다른 모델이 활용함
  - 컴퓨터 비전 분야에서 많이 사용됨
    - 학습된 모델을 사용해 이미지의 특성을 추출하고 이런 특성들을 객체 탐지나 이미지 분할 같은 다른 컴퓨터 비전 작업에 활용했음

- 파인튜닝은 모델 학습 과정의 일부, 사전 학습의 확정
  - 사전 학습 이후에 모델을 추가로 학습시키는 과정

- 모델 학습 : 보통 자기 지도 학습 방식의 사전 학습에서 시작함
  - 자기 지도 학습을 통해 모델은 레이블이 없는 대량의 데이터에서 학습할 수 있음
  - 언어 모델의 경우, 자기 지도 학습 데이터는 주로 주석이 필요 없는 텍스트 시퀀스
  - 자기 지도 파인튜닝 : 지속적 사전 학습(Continued pre-training)
  - 인필링 파인튜닝 : 지도 파인튜닝을 통해 다음 토큰을 예측하거나 빈칸을 채우도록 모델을 파인튜닝할 수도 있음
    - 텍스트 편집 및 코드 디버깅 같은 작업에 특히 유용함

- 자기 지도 학습을 통해 모델은 세상에 대한 방대한 지식을 얻지만 이 지식을 사용자가 원하는 특정 작업에 바로 활용하기는 어려움
  - 지도 파인튜닝은 사용자가 원하는 특정 작업에 활용가능하게 하고 사람의 선호와 일치할 수 있게 격차를 줄이는 역할을 함
    - 고품질 주석 데이터를 활용해 모델을 사람의 사용 방식과 선호도에 맞게 조정함

- 지도 파인튜닝 : (입력, 출력) 쌍으로 모델을 학습함
  - 입력은 지시가 될 수 있고, 출력은 응답이 될 수 있음

- 모델은 또한 강화 학습을 통해 사람의 선호도를 최대화하는 응답을 생성하도록 파인튜닝할 수 있음
  - 선호도 파인튜닝 : (지시, 선호 응답, 비선호 응답) 형식의 비교 데이터가 필요함
 
- 컨텍스트 길이를 늘리기 위해 모델을 파인튜닝하는 것도 가능함 : 롱 컨텍스트 파인튜닝
  - 위치 임베딩 조정 같은 모델 구조의 수정이 필요함
  - 롱 시퀀스는 토큰의 가능한 위치가 더 많다는 뜻
    - 위치 임베딩이 이를 처리할 수 있어야 함
  - 다른 파인튜닝 기법보다 롱 컨텍스트 파인튜닝은 더 까다로움
    - 결과로 얻어진 모델은 숏 시퀀스에서 성능이 오히려 떨어질 수 있음
   

## 7.2 파인튜닝이 필요한 경우
- 프롬프트 기반 방법과 비교 시 파인튜닝은 많은 데이터와 고사양 하드웨어를 요구할 뿐만 아니라 이를 다룰 ML 전문가도 필요함.
- 보통 프롬프트 기반 방법을 충분히 시도한 후에 파인튜닝을 시도하는 것이 일반적
  - 두 방법이 양자택일 관계는 아님
  - 두 접근법을 함께 활용해야 하는 경우가 많음
 
### 7.2.1 파인튜닝을 해야 하는 이유
- 주요 목적 : 일반 능력과 특정 작업 수행 능력을 모두 향상시키는 데 있음
  - JSON이나 YAML 같은 특정 구조의 출력을 생성할 때 파인튜닝이 효과적
  - 사용하려는 모델이 특정 작업에서 충분히 학습되지 않았다면 해당 작업에 관련된 자체 데이터로 파인튜닝하는 것이 특히 효과적
 
- 흥미로운 활용 사례 중 하나 : 편향 완화
  - 기본 모델이 학습 데이터의 특정 편향을 반영한다면 파인 튜닝 과정에서 신중하게 선별된 데이터를 사용해 이런 편향을 상쇄할 수 있음

- 큰 모델을 파인튜닝해 더 개선할 수도 있지만 작은 모델을 파인튜닝하는 경우가 훨씬 더 일반적
  - 작은 모델은 메모리 요구량이 적어 파인튜닝하기 쉽고 운영 환경에서도 더 저렴하고 빠르게 사용할 수 있음
  - 큰 모델이 생성한 데이터로 작은 모델을 학습시켜 마치 큰 모델처럼 작동하게 만드는 방식이 흔히 쓰임
    - 해당 방식: 큰 모델의 지식을 더 작은 모델에 효율적으로 전달하는 과정 => 증류라고 불림
      - 증류라고 불리는 이유 : 복잡한 혼합물에서 핵심 성분을 추출하는 '증류' 과정과 유사
   
### 7.2.2 파인튜닝을 하지 말아야 하는 이유
- 파인튜닝으로 모델 성능을 향상시킬 수 있지만 잘 작성된 프롬프트와 컨텍스트도 비슷한 효과를 냄
- 특정 작업을 위해 모델을 파인튜닝하면 그 작업에서는 성능이 햐앙도리 수 있지만 다른 작업에서는 오히려 성능이 떨어질 수 있음
  - 다양한 프롬프트를 사용해야 하는 애플리케이션에서 문제
- 프로젝트를 실험을 막 시작하는 단계면 파인튜닝은 가장 먼저 시도할 방법이 아님(파인튜닝은 초기 투자가 크고 지속적인 관리가 필요)
   1. 데이터가 필요
      - 주석이 달린 데이터는 수동으로 모으는 데에 시간이 걸리고 비용이 많이 듬
   2. 파인튜닝은 모델 학습 방법에 대한 지식이 필요함
      - 파인튜닝할 모델을 선택하기 위해 기본 모델을 평가해야 함
        - 필요와 자원에 따라 선택지가 제한될 수 있음
      - 파인튜닝 프레임워크와 API가 실제 파인튜닝 과정의 많은 단계를 자동화할 수 있음
        - BUT, 여전히 조정할 수 있는 다양한 학습 파라미터를 이해하고 학습 과정을 모니터링하여 문제가 발생햇을 때 디버깅할 줄 알아야 함.
   3. 파인튜닝된 모델을 얻은 후에는 이를 서빙하는 방법을 알아내야 함
      - 직접 호스팅할 것인지 아니면 API 서비스를 사용할지 정해야 함
- 더 중요한 것은 모델을 모니터링하고 유지보수하며 업데이트하기 위한 정책과 예산을 수립해야 한다는 점
  - 파인튜닝된 모델을 계속 개선하는 동안에도 새로운 기본 모델들이 빠른 속도로 개발되고 있음
  - 이런 기본 모델들은 파인튜닝된 모델을 개선할 수 있는 속도보다 더 빠르게 발전할 수 있음
    - 과연 성능이 얼마나 좋아져야 새로운 모델로 갈아탈 만한 가치가 있을까? 등등 고민하는 과정이 필요
- 프롬프팅부터 시작하는 것이 좋음
  - 프롬프팅만으로 부족할 때만 더 고급 솔루션을 고려
  - 다양한 프롬프트를 철저히 테스트했는지 확인해야 함
  - 모델의 성능은 프롬프트에 따라 크게 달라질 수 있음

- **도메인 특화 작업 파인튜닝**
  - 범용 모델이 특정 도메인 작업에 잘 작동하지 않으므로 일단 특정 작업에 맞게 모델을 파인튜닝하거나 학습시켜야 한다는 주장을 조심해야 함
    - 범용 모델의 모든 능력이 더 강력해질수록 특정 도메인 작업에도 더 능숙해져 도메인 특화 모델보다 오히려 더 나은 성능을 보일 수 있음


- 파인튜닝과 프롬프팅 실험 모두 체계적인 접근 방식이 필요함
  - 프롬프트 캐싱이 도입되기 전에는 파인튜닝의 큰 장점 중 하나가 토큰 사용을 최적화하는 데 도움이 된다는 것
  - 프롬프트에 예시를 많이 넣을수록 모델이 처리해야할 입력 토큰이 늘어나 처리 속도가 느려지고 비용도 증가함
  - 매번 프롬프트에 예시를 포함하는 대신 이런 예시로 모델을 파인튜닝하면 더 짧은 프롬프트만으로도 같은 결과를 얻을 수 있음
- 반복적인 프롬프트 부분을 저장했다가 재사용할 수 있는 프롬프트 캐싱 기술이 도입되면서 파인튜닝의 장점은 크게 줄었음
  - 여전히 프롬프트와 함께 사용할 수 있는 예시 수는 최대 컨텍스트 길이에 제한을 받는 반면 파인튜닝에서는 활용할 수 있는 예시 수에 제한이 없음.
 

### 7.2.3 파인튜닝과 RAG
- 프롬프팅을 통해 모델의 성능을 최대한 끌어올렸따면 다음으로 RAG를 적용할지 파인튜닝을 할지 고민하게 됨
- 모델이 정보가 부족해서 오답을 내놓은다면 관련 정보 소스에 접근할 수 잇게 해주는 RAG 시스템이 효과적
  - 정보 기반 오류 : 출력 내용이 사실과 다르거나 정보가 오래된 경우에 발생함
  - 정보 부족으로 인한 오류가 발생하는 두 가지 상황
    - 모델이 정보를 가지고 있지 않은 경우
      - 공개 모델은 사용자나 조직의 비공개 정보를 알지 못하는 경우가 많음
      - 모델이 정보를 모르면 그 사실을 알려주거나 응답을 지어내게 됨
    - 모델의 정보가 오래된 경우
      - 모델의 지식 기준 날짜가 최신보다 이전임
- <Fine-Tuning or Retrieval?> 논문은 시사 문제에 관한 질의와 최신 정보가 필요한 작업에서는 RAG가 파인튜닝된 모델보다 더 좋은 성능을 보인다는 점을 입증햇음
  - 오바디아 등 논문 내의 선별한 시사 문제에 대한 질의-응답 작업에서 RAG가 파인튜닝보다 더 뛰어난 성능을 보였음
  - 기본 모델을 사용한 RAG가 파인튜닝된 모델을 사용한 RAG보다도 더 나은 결과를 냈음
    - 파인튜닝이 특정 작업의 성능은 높여줄 수 있지만 다른 영역에서는 오히려 성능이 떨어질 수 있음

- 모델의 행동 방식에 문제가 있다면 파인튜닝이 도움이 될 수 있음
  - 행동 방식의 문제 중 하나는 모델의 출력이 사실 관계는 맞지만 요청한 작업과 전혀 무관한 경우
    - EX) 엔지니어링 팀에 제공할 소프트웨어 프로젝트의 기술 명세서를 만들어달라고 모델에 요청했음
    - 생성된 명세서가 정확하더라도 팀이 실제로 필요로 하는 세부 사항이 없을 수 있음
  - 모델이 원하는 출력 형식을 제대로 따르지 못하는 경우
    - EX) 모델에게 HTML 코드 작성을 요청했는데 생성된 코드가 제대로 작동하지 않는다면 모델이 학습과정에서 HTML에 충분히 노출되지 않았기 때문
    - 파인튜닝을 통해 더 많은 HTML 코드 예시를 학습시켜 이 문제를 해결할 수 있음
   
- 시맨틱 파싱 : 모델이 정해진 형식에 맞춰 출력을 생성하는 능력이 작업의 성패를 좌우함
  - 시맨틱 파싱 작업에는 종종 파인튜닝이 필요함 : 자연어를 JSON 같은 구조화된 형식으로 변환하는 과정
 
- 요약 : **파인튜닝은 형식을 위한 것이고 RAG는 사실을 위한 것**
  - RAG 시스템 : 모델에 외부 지식을 제공해 더 정확하고 유익한 응답을 만들 수 있게 함
    - 모델의 환각 현상을 줄이는 데 도움이 될 수 있음
  - 파인튜닝 : 모델이 특정 문구와 스타일을 이해하고 따르는 데 도움을 줌
    - 고품질 데이터로 파인튜닝 진행 시 모델의 환각 현상을 줄일 수도 있지만 데이터 품질이 낮으면 오히려 더 심하게 만들 수 있음
   
- 모델이 정보와 행동 측면 모두에 문제가 있다면 RAG부터 시작하는 것이 좋다.
  - RAG는 일반적으로 학습 데이터를 수집하거나 파인튜닝된 모델을 별도로 운영할 필요가 없어 더 쉽게 접근할 수 있음
  - RAG 구현 시 벡터 데이터베이스로 바로 뛰어들기보다 BM25 같은 단순한 키워드 기반 검색부터 시작하는 것이 좋음

- 모델 조정의 전체적인 과정
  - 우선 프롬프트만으로 모델이 원하는 작업을 수행하도록 해봄
  - 프롬프트에 더 많은 예시를 추가한다(사용 사례에 필요한 예시의 수 : 1~50개 정도)
  - 모델이 정보 부족으로 자주 오류를 보인다면 관련 정보를 제공할 수 있는 데이터 소스에 연결함
    - RAG를 시작할 때는 키워드 기반 검색 같은 기본적인 검색 방법부터 시작하는 것이 좋음
  - 모델의 오류 유형에 따라 다음 단계 중 하나를 시도해 볼 수 있음
    - 모델이 계속해서 정보 관련 오류를 보인다면 임베딩 기반 검색 같은 고급 RAG 방법을 시도
    - 모델이 관련 없는 내용을 생성하거나 형식이 잘못되거나 안전하지 않은 응답을 생성하는 등의 행동 측면의 문제가 지속된다면 파인튜닝을 고려함
      - 임베딩기반 검색은 파이프라인에 추가 구성 요소를 도입해 추론 과정의 복잡성을 높이는 반면, 파인 튜닝은 모델 개발 과정은 복잡해지지만 추론 과정은 그대로 유지됨
  - 더 큰 성능 향상을 위해 RAG와 파인튜닝을 함께 활용함
